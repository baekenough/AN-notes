## Gemini CLI ya tiene un modelo de seguridad bastante claro

Todavía mucha gente piensa en Gemini CLI como "un chat en la terminal". La documentación actual muestra algo más serio:

- **trusted folders** para decidir qué workspaces desbloquean capacidades completas
- **sandboxing** para aislar acciones de riesgo
- **checkpointing** para volver atrás después de cambios en archivos

Juntas, estas funciones hacen que Gemini CLI sea mucho más utilizable en repositorios reales.

## Trusted folders: protegerte de workspaces riesgosos

Los trusted folders vienen **desactivados por defecto**. Cuando los activas, Gemini CLI pregunta si debe confiar en una carpeta antes de cargar configuración específica del proyecto.

Actívalo en tu configuración de usuario:

```json
{
  "security": {
    "folderTrust": {
      "enabled": true
    }
  }
}
```

### Opciones del diálogo de confianza

- **Trust folder** — confiar solo en el proyecto actual
- **Trust parent folder** — confiar en todo el árbol del directorio padre
- **Don't trust** — ejecutar en modo seguro restringido

Las decisiones se guardan en:

```text
~/.gemini/trustedFolders.json
```

## Qué cambia en un workspace no confiable

La documentación oficial indica que el modo no confiable desactiva o restringe, entre otras cosas:

- `.gemini/settings.json` del workspace
- carga de `.env` del proyecto
- instalar / actualizar / desinstalar extensiones
- auto-acceptance de herramientas
- carga automática de memoria desde settings locales
- conexión a servidores MCP
- comandos personalizados

Si trabajas con repos de clientes, demos o código de terceros, esta función importa mucho.

## Sandboxing: aislar acciones con efectos laterales

Gemini CLI puede ejecutar modificaciones de archivos y comandos shell dentro de un sandbox en lugar de hacerlo en tu entorno host.

```bash
gemini --sandbox
```

También puede controlarse con `GEMINI_SANDBOX` y otras opciones de configuración.

### Imagen de sandbox específica por proyecto

La documentación soporta un Dockerfile personalizado en:

```text
.gemini/sandbox.Dockerfile
```

Eso ayuda cuando un proyecto necesita binarios o dependencias concretas y no quieres contaminar la máquina local.

## Checkpointing: crear un punto de retorno antes de escribir archivos

Checkpointing es una función separada y viene **apagada por defecto**.

Actívala en `settings.json`:

```json
{
  "general": {
    "checkpointing": {
      "enabled": true
    }
  }
}
```

Cuando está activa, Gemini CLI crea un checkpoint antes de ejecutar herramientas que escriben archivos.

Ese checkpoint incluye:

1. un shadow Git snapshot
2. el historial de conversación
3. la tool call que iba a ejecutarse

### Restaurar con `/restore`

```text
/restore
/restore <checkpoint_file>
```

Para sesiones con muchos cambios, esta es una de las funciones más valiosas del producto.

## Cómo lo usaría en la práctica

### Repo propio y confiable

- confiar la carpeta
- trabajar normalmente en modo interactivo
- activar checkpointing si habrá muchas ediciones

### Repo desconocido de cliente

- activar trusted folders globalmente
- no confiar de inmediato
- preferir sandboxing para acciones con riesgo

### Automatización o ráfagas de cambios generados

- ejecutar en sandbox
- mantener checkpointing activado
- revisar diffs antes de aceptar ediciones amplias
