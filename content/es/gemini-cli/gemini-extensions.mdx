## Las extensiones son más que "plugins"

La documentación actual describe las extensiones como paquetes que pueden incluir:
- prompts
- servidores MCP
- comandos personalizados
- hooks
- sub-agentes
- **agent skills** (habilidades de agente)

## Piensa en capas

### Capa 1: `GEMINI.md`
Para instrucciones estables y reglas del equipo.

### Capa 2: MCP
Para dar acceso a bases de datos, APIs o herramientas externas (Jira, Confluence, etc.).

### Capa 3: Extensions y Skills
Para **empaquetar y compartir** workflows complejos. Aquí es donde brillan las **Agent Skills**.

## El Poder de las Agent Skills

Una de las características más singulares de Gemini CLI es poder activar "modos expertos" mediante la herramienta `activate_skill`.

- **Skills incorporadas**: Permiten sobreescribir la mente del agente para enfocarse en un objetivo. Por ejemplo, `security-review` (mentalidad hacker), `code-review` (ingeniero senior) o `web-clone` (clonación visual de UI).
- **Crear tus propias skills**: Con `skill-creator`, puedes interactuar con el agente para crear flujos únicos de tu empresa (ej: "checklist de QA interno") y guardarlo permanentemente como una habilidad compartible en tu equipo.

## Qué puedes gestionar hoy

Dentro del CLI interactivo:
```text
/extensions list
```

Desde la terminal:
```bash
gemini extensions list
# Instalar desde GitHub
gemini extensions install https://github.com/gemini-cli-extensions/workspace
```

## Un patrón sensato para equipos

1. Poner reglas del repo en `GEMINI.md`.
2. Conectar uno o dos MCP esenciales.
3. Formalizar procesos repetitivos creando habilidades con `skill-creator`.
4. Empaquetarlo como extension cuando el flujo sea estándar.

## Recomendación para este proyecto

Si AN-Con crece, una buena extension de Gemini CLI probablemente agruparía:
- comandos de revisión de paridad
- helpers para validar frontmatter MDX
- una Skill propia (`locale-reviewer`) para tono y localización
- un MCP de búsqueda documental
