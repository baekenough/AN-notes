## ¿Qué Son los Subagentes?

Claude Code puede generar subagentes especializados — instancias separadas de Claude que trabajan en tareas enfocadas en paralelo. Piénsalo como Claude delegando trabajo a un equipo.

## Cuándo Usar Subagentes

| Escenario | Agente Único | Subagentes |
|-----------|-------------|-----------|
| Corregir un typo | ✅ | Excesivo |
| Revisar 3 archivos | ✅ | ✅ (más rápido) |
| Refactorizar 20 archivos | Muy lento | ✅ |
| Investigar + implementar | ✅ | ✅ (paralelo) |
| Depuración entre módulos | Pierde contexto | ✅ |

## El Patrón Orquestador

La conversación principal actúa como orquestador — planifica el trabajo y delega a especialistas:

```
Tú: "Revisa el módulo de autenticación por problemas de seguridad"

Claude (orquestador):
  → Genera Agente 1: Revisar auth/login.ts por riesgos de inyección
  → Genera Agente 2: Revisar auth/session.ts por manejo de tokens
  → Genera Agente 3: Revisar auth/middleware.ts por riesgos de bypass
  → Agrega hallazgos en un único reporte
```

## Ejemplo Práctico: Refactorización Grande

```
Migra todas las rutas API del patrón Express middleware a Next.js Route Handlers.

Hay 15 rutas en src/api/. Usa agentes paralelos —
uno por grupo de rutas (auth, users, products, orders, analytics).

No modifiques las funciones utilitarias compartidas en src/api/utils/.
```

## Agent Teams (Experimental)

La función Agent Teams de Claude Code habilita comunicación peer-to-peer entre agentes, ideal para:
- Ciclos de revisión de código → corrección → re-revisión
- Depuración compleja entre módulos
- Investigación que requiere discusión y síntesis

## Agentes en Segundo Plano

Ejecutar agentes en segundo plano te permite seguir trabajando en otras cosas sin esperar. Es especialmente útil para tareas largas.

```
Ejecuta la refactorización grande en segundo plano.
Avísame cuando termine — seguiré con otras cosas mientras tanto.
```

Casos ideales para ejecución en segundo plano:
- **Refactorización a gran escala** — convertir decenas de archivos
- **Ejecución de tests largos** — correr la suite de tests completa
- **Revisión de código** — analizar todos los archivos de un PR grande
- **Análisis de dependencias** — recorrer el árbol de dependencias del proyecto

Cuando el agente termina, recibes una notificación con los resultados. Puedes retomar tu trabajo cuando quieras.

## Aislamiento con Git Worktrees

Que cada agente trabaje en su propio git worktree permite experimentar sin tocar la rama principal.

```
Haz que cada agente trabaje en su propio worktree:
- Agente 1: Refactorizar módulo de auth en la rama feature/auth-refactor
- Agente 2: Migrar APIs en la rama feature/api-migration
- Agente 3: Mejorar cobertura de tests en la rama feature/test-coverage

Cada uno trabaja de forma independiente y entrega resultados en su propia rama.
```

Ventajas del aislamiento con worktrees:
- **Experimentación segura** — si falla, solo borras el worktree
- **Trabajo en ramas en paralelo** — desarrolla varias funcionalidades a la vez
- **Sin interferencias** — los agentes no se pisan entre sí
- **Rollback sencillo** — descarta una rama si el resultado no convence

## Patrones de Especialización de Agentes

Asignar a cada agente un rol experto específico produce análisis mucho más profundos que un único agente generalista.

```
Analiza el proyecto completo con tres agentes especializados:

Agente de Seguridad:
  - Verificar contra OWASP Top 10
  - Detectar riesgos de SQL injection, XSS y CSRF
  - Revisar lógica de autenticación y autorización

Agente de Rendimiento:
  - Detectar patrones de consultas N+1
  - Identificar posibles fugas de memoria
  - Detectar re-renders innecesarios y cálculos costosos

Agente de Cobertura de Tests:
  - Medir la cobertura de código actual
  - Identificar casos límite no cubiertos
  - Listar módulos sin tests de integración
```

Cada agente se centra en su dominio y produce resultados que un único agente repartido entre las tres áreas no alcanzaría.

## Escenario Real: Frontend + Backend al Mismo Tiempo

Al añadir una nueva funcionalidad, puedes construir el frontend, el backend y la base de datos a la vez:

```
Añade la funcionalidad "Perfil de Usuario". Que tres agentes trabajen a la vez:

Agente 1 (Backend): Implementar endpoints GET/PUT /api/users/:id
Agente 2 (Frontend): Construir componentes React ProfilePage y ProfileForm
Agente 3 (Base de datos): Migración para añadir columnas bio y avatar_url a la tabla users

Los tipos compartidos están en src/types/user.ts.
Cuando los tres terminen, propón un plan de integración.
```

Con tres agentes en paralelo, el tiempo total se acerca al del agente más lento, en lugar de ser la suma de los tres.

## Cuándo los Agentes No Valen la Pena

Más agentes no siempre es mejor. Ten en cuenta estos puntos:

- **Las tareas simples se vuelven más lentas** — corregir un typo o editar un archivo es más rápido con un solo agente. Crear agentes tiene su coste de arranque.
- **El contexto es independiente** — cada agente tiene su propia ventana de contexto. Compartir estado entre agentes requiere archivos o una lista de tareas compartida.
- **El coste se multiplica** — número de agentes × llamadas a la API = mayor gasto. Úsalos cuando el paralelismo realmente lo justifique.
- **Más de 4 agentes puede perder eficiencia** — Claude Code recomienda un máximo de 4 agentes simultáneos. Por encima de eso, la sobrecarga de coordinación crece.

## Claude Code SDK para Orquestación Programática

Claude Code se puede usar como paquete npm (`@anthropic-ai/claude-code`), lo que permite construir flujos de orquestación personalizados en TypeScript/JavaScript.

Uso básico:

```typescript
import { claude } from "@anthropic-ai/claude-code";

// Ejecución simple de prompt
const result = await claude({
  prompt: "Analyze src/auth/ for security vulnerabilities",
  options: {
    maxTurns: 10,
    allowedTools: ["Read", "Glob", "Grep"],
  }
});

console.log(result.text);
```

Ejemplo de orquestación multi-agente:

```typescript
// Revisión de código en paralelo por módulos
const modules = ["src/auth/", "src/api/", "src/database/"];

const reviews = await Promise.all(
  modules.map(mod =>
    claude({
      prompt: `Review ${mod} for security issues. Focus on injection risks and auth bypasses.`,
      options: { allowedTools: ["Read", "Glob", "Grep"] }
    })
  )
);

// Agregar resultados
const summary = await claude({
  prompt: `Synthesize these security reviews into a single report:\n${reviews.map(r => r.text).join("\n---\n")}`,
});
```

Casos de uso: pipelines de CI personalizados, auditorías de código automatizadas, procesamiento por lotes en bases de código grandes.

## Modo Headless para Automatización

`claude -p "prompt"` se ejecuta sin conversación interactiva. Combinado con scripts de shell, permite una automatización potente:

```bash
#!/bin/bash
# Auditoría de seguridad en paralelo por servicio

services=("auth-service" "payment-service" "user-service")

for service in "${services[@]}"; do
  claude -p "Audit $service/ for OWASP Top 10 vulnerabilities. Output as markdown." \
    --model haiku \
    --output-format text \
    > "reports/${service}-audit.md" &
done

wait
echo "All audits complete"
```

Opciones útiles:
- `--output-format json` para salida parseable por máquinas
- Piping por stdin: `git diff HEAD~5 | claude -p "Summarize the changes in these commits"`
- Variable de entorno `ANTHROPIC_API_KEY` para sistemas CI/CD

El modo headless es especialmente útil para integrar Claude Code como herramienta de automatización en GitHub Actions, Jenkins y otros sistemas de CI.

## Estrategias de Optimización de Costos

Cada subagente crea un nuevo contexto de Claude, lo que implica costos adicionales de API. Estas son las estrategias para minimizar el gasto:

| Estrategia | Cómo | Ahorro |
|------------|------|--------|
| Mezcla de modelos | haiku para búsqueda/lectura, sonnet para implementación | ~60% en tareas de lectura intensiva |
| Alcance limitado | Alcance reducido = menos turnos = menos costo | ~40% por agente |
| Caché de resultados | Almacenar salidas, reutilizar entre ejecuciones | Variable |
| Agrupación en lotes | Agrupar archivos relacionados en menos agentes | ~30% vs un agente por archivo |

Anti-patrón: crear un agente por archivo para 50 archivos. En su lugar, crea agentes por módulo.

Punto óptimo: 2-4 agentes para la mayoría de tareas, cada uno manejando un grupo lógico de archivos relacionados.

## Escenario Real: Sprint de Funcionalidad Full-Stack

Recorramos un escenario multi-agente realista:

```
Tarea: Añadir funcionalidad de "notificaciones de usuario" a una app full-stack

Plan de orquestación de agentes:
├── Agente 1 (Backend): Diseñar esquema de notificaciones, crear endpoints API
│   Alcance: src/api/notifications/, prisma/schema.prisma
│
├── Agente 2 (Frontend): Construir componente NotificationBell, página de lista de notificaciones
│   Alcance: src/components/notifications/, src/pages/notifications/
│
├── Agente 3 (Infraestructura): Configurar servidor WebSocket, Redis pub/sub
│   Alcance: src/services/websocket/, docker-compose.yml
│
└── Agente 4 (Testing): Escribir tests de integración para el flujo de notificaciones
    Alcance: tests/notifications/
    Dependencias: Espera a que los Agentes 1-3 completen

Ejecución: Agentes 1-3 en paralelo → Agente 4 después
Tiempo total: ~5 min (vs ~15 min secuencial)
```

Punto clave: define las interfaces compartidas (`types/notification.ts`) antes de generar los agentes. Cada agente tiene acceso de lectura a los tipos compartidos pero solo escribe en su propio alcance.

## Comparación con Codex

GPT Codex ejecuta cada tarea como un único agente en sandbox. No hay orquestación multi-agente, pero el modelo de sandbox da aislamiento completo de contenedor por tarea.

| Característica | Subagentes Claude Code | Tareas Codex |
|---------------|----------------------|-------------|
| Paralelismo | Hasta 4 concurrentes | Tareas concurrentes posibles |
| Comunicación | Contexto compartido, mensajería | Independiente, aislado |
| Mejor para | Trabajo complejo e interconectado | Tareas independientes bien definidas |
