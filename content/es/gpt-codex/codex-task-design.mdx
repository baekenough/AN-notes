## La regla central: quitar adivinanza

Codex es fuerte ejecutando.
Se vuelve más débil cuando lo obligas a inferir qué significa éxito.

Un buen prompt no tiene que ser elegante; tiene que ser lo bastante específico para que el modelo invente menos.

## La estructura mínima útil

Las best practices de Codex recomiendan prompts con forma de issue.

```text
Goal:
¿Qué debe cambiar?

Context:
¿Qué archivos, errores o patrones importan?

Constraints:
¿Qué no debe romperse o cambiar?

Done when:
¿Qué demuestra que la tarea terminó?
```

Parece simple, pero funciona.
Sin Goal no hay blanco. Sin Done when no hay línea de llegada.

## Prompt malo vs prompt mejor

Demasiado vago:

```text
mejora el flujo del carrito
```

Mucho mejor:

```text
Goal:
Agregar optimistic updates al control de cantidad del cart drawer.

Context:
- La UI vive en src/components/cart/
- La server action vive en src/app/actions/cart.ts
- Sigue el patrón de loading ya usado en wishlist

Constraints:
- no cambies el shape del API público
- no agregues una nueva librería de estado

Done when:
- los cambios de cantidad se sienten inmediatos
- hay rollback limpio si falla
- pasan lint, build y los tests relevantes
```

## Dimensiona la tarea alrededor de un loop de verificación

Una regla práctica no es “una feature”, sino **un loop de verificación revisable**.

Buenas tareas:

- un refactor acotado en una carpeta
- un bug reproducible
- updates de docs ligados a un cambio puntual de API

Malas tareas:

- rediseñar auth, onboarding y billing juntos
- modernizar toda la app
- limpiar toda la deuda técnica

Si un solo loop de lint/build/test no puede decirte si la tarea está sana, probablemente es demasiado grande.

## Algunos trabajos deben empezar solo con plan

Conviene pedir plan primero cuando:

- el cambio cruza varios dominios
- pesan más las decisiones de arquitectura que la velocidad de coding
- el rollback sería costoso
- hay varias implementaciones razonables

Un buen arranque es:

```text
No implementes todavía. Propón primero un plan.
Incluye riesgos, archivos tocados y pasos de verificación.
```

## Los ejemplos locales ganan a los consejos abstractos

Codex responde mejor a “sigue este archivo” que a “usa nuestra arquitectura habitual”.

Mejores anclas:

- “sigue el patrón de src/features/profile/actions.ts”
- “usa el formato ya presente en docs/releases/”
- “reutiliza el patrón de nombres de variantes del Button”

Cuanto más concreto sea el punto de referencia, menos tendrá que adivinar Codex.

## Las restricciones son una herramienta de calidad

Las restricciones no son fricción. Son dirección.

Restricciones de alto valor:

- no agregar dependencias
- preservar el shape del API público
- quedarse dentro de un directorio
- no tocar archivos generados
- mantener el diff pequeño

## Cuándo ir más allá de un prompt simple

Si el mismo tipo de tarea se repite, quizá exista una superficie mejor:

- workflow repetido → [Skills](./codex-skills)
- hace falta contexto externo → [MCP](./codex-mcp)
- la exploración o implementación paralela ayuda → pedir subagents explícitamente

La documentación de Subagents aclara que Codex solo los lanza cuando lo pides, y que heredan la sandbox policy actual. El paralelismo ayuda, pero no es gratis.

## Usa Best-of-N cuando haya incertidumbre

OpenAI destaca Best-of-N para trabajos más difíciles.
Sirve cuando:

- el naming o la estructura importan mucho
- varias soluciones parecen razonables
- quieres comparar opciones antes de mergear

## Checklist rápido antes de lanzar la tarea

Antes de mandar una tarea no trivial, revisa esto:

- ¿el objetivo es singular?
- ¿nombraste archivos o carpetas relevantes?
- ¿listaste restricciones importantes?
- ¿definiste éxito en términos observables?
- ¿el cambio cabe en un loop de review?

El gran task design en Codex no es un truco oculto de prompting.
Es la disciplina de dejar la tarea clara incluso para otro ingeniero.
