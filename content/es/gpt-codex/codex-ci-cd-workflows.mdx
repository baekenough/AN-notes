## La palanca real aparece en el trabajo repetido

Una tarea exitosa con Codex sirve.
Un workflow repetible con Codex vale mucho más.

Ahí es donde review loops, mantenimiento, sincronización de docs, refactors seguros y automatización acotada empiezan a multiplicar valor.

## Haz de `/review` un hábito

Los docs de Review explican que el review pane refleja el estado de tu repositorio Git.
Por defecto se enfoca en **uncommitted changes**, y puedes cambiar el alcance a:

- todos los cambios de la rama contra la base
- solo los cambios del último turno
- staged vs unstaged localmente

Eso importa porque transforma la salida de Codex otra vez en un artefacto de ingeniería inspeccionable.

Un loop corto y fuerte es:

1. implementación
2. lint / build / tests
3. `/review`
4. stage, revert o iteración

## ¿Qué cambios requieren la review más fuerte?

La review humana es especialmente importante para:

- cambios de contrato de API
- schemas o migraciones
- auth, billing o lógica de seguridad
- copy y comportamiento user-facing
- renames o borrados grandes

Codex puede producir un draft rápido, pero el juicio irreversible sigue siendo humano.

## Los worktrees dan velocidad y aislamiento al mismo tiempo

La documentación de Worktrees explica que Codex puede correr múltiples tareas independientes en paralelo dentro del mismo proyecto sin pisarse.
En repos Git, las automations también pueden correr en background sobre worktrees dedicados.

Eso sirve porque:

- reduce conflictos con tu working tree principal
- aísla runs experimentales
- hace más segura la automatización recurrente

Un worktree es al mismo tiempo una feature de velocidad y de blast radius.

## Usa automations solo cuando el workflow ya sea estable

La documentación de Automations describe ejecuciones recurrentes en background donde eliges project, prompt, cadence y execution environment. También pueden invocarse skills dentro de esas runs.

Buenos candidatos para automation:

- resúmenes de commits recientes
- borradores de release notes
- chequeos de docs drift
- escaneos de bugs probables
- checks estáticos de bajo riesgo

Malos candidatos:

- trabajo de producto que requiere juicio constante
- flujos con acceso amplio a secretos o producción
- pasos de despliegue con alto costo de fallo

## Cómo se ve Codex dentro de CI/CD en la práctica

El patrón más práctico casi nunca es “dejar que Codex mergee solo”.
Normalmente es **automatizar el trabajo repetido de ingeniería antes del punto de decisión humana**.

Ejemplos:

- redactar una primera pasada de review de PR
- convertir un label de issue en una tarea acotada
- generar un primer draft de updates grandes de docs
- detectar zonas candidatas a falta de tests

El valor está menos en la autonomía total y más en acelerar el trabajo repetible antes de la review.

## Skill + automation + worktree + review

Una vez que el workflow es estable, esta combinación es poderosa:

- skill = procedimiento reutilizable
- automation = scheduling
- worktree = aislamiento
- `/review` = inspección final

Cuando estas cuatro piezas encajan, Codex se parece menos a un asistente one-off y más a un sistema de ingeniería repetible.

## Checklist final de operación

Antes de automatizar o productivizar un workflow, pregúntate:

- ¿el shape de la tarea es lo bastante estrecho?
- ¿sandbox y approvals coinciden con el riesgo?
- ¿los checks son mecánicos y fiables?
- ¿un humano puede revisar el resultado con claridad?
- ¿la run queda aislada del working tree principal?

Los equipos que usan bien Codex en el tiempo no son los que tienen los permisos más amplios.
Son los que mantienen límites revisables y repetibles.
