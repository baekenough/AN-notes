## El sandbox es seguridad y también UX

Si el límite es demasiado estricto, todo se convierte en approvals.
Si es demasiado amplio, un error pequeño puede costar caro.

La meta real es esta:
**dejar que Codex haga trabajo normal sin frenarse todo el tiempo, mientras las acciones riesgosas siguen bajo control humano.**

## Mantén separados local y cloud en tu cabeza

### Codex local
Corre en tu máquina, con sandbox y approval policy definidos por ti.

### Codex cloud
Corre en un contenedor administrado por OpenAI, donde setup scripts, variables de entorno y política de internet importan más.

Mezclar ambos modelos es una fuente clásica de confusión.

## Los tres sandbox modes locales más comunes

La documentación de Sandboxing los resume así:

| Modo | Significado | Cuándo usarlo |
|------|-------------|---------------|
| `read-only` | inspección; editar o ejecutar pide approval | auditoría, exploración, review segura |
| `workspace-write` | edita dentro del workspace y corre comandos rutinarios | desarrollo diario normal |
| `danger-full-access` | elimina restricciones del sandbox | automatización rara y de mucha confianza |

OpenAI describe `workspace-write` como el modo local de baja fricción por defecto.

## La approval policy define cuándo se detiene Codex

Los docs de config muestran una combinación muy común:

```toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"
```

Funciona bien porque:

- el trabajo normal dentro del repo sigue avanzando
- los cruces de frontera todavía piden permiso

Para la mayoría de los equipos, ese es el default práctico.

## Writable roots importa más de lo que parece

Mucha gente mira solo el sandbox mode y olvida writable roots.
En la práctica, ahí vive bastante riesgo accidental.

Buenos defaults:

- el propio repositorio
- quizá un directorio temporal

Defaults riesgosos:

- todo tu home
- múltiples proyectos sin relación

La pregunta no es solo “¿puede escribir Codex?”, sino **¿exactamente dónde puede escribir?**

## En cloud, setup es el contrato

Cloud Environments describe un flujo parecido a este:

1. crear el contenedor y hacer checkout del repo
2. ejecutar el setup script
3. aplicar la política de internet
4. dejar que el agente edite y verifique

Dos detalles importan mucho:

- **los setup scripts corren con internet**
- **el internet en la agent phase está apagado por defecto**

Por eso instalación de dependencias, codegen y bootstrap de tests conviene moverlos a setup.

## Qué sí y qué no meter en setup

Buenos candidatos:

- instalar dependencias
- fijar versiones de runtime o tooling
- preparar datos de test
- codegen
- preparación de builds nativos

Malos candidatos:

- implementar toda la feature
- pasos que requieren juicio humano repetido

Setup debe volver reproducible el entorno, no resolver la tarea en silencio.

## Review y sandbox deberían pensarse juntos

La documentación de Review explica que el panel se enfoca por defecto en **uncommitted changes**, y puede cambiar a diff contra base branch o last-turn changes.
Eso importa porque permisos más amplios implican una necesidad mayor de review.

En la práctica:

- sandbox más estricto = más interrupciones, menor blast radius
- sandbox más amplio = menos interrupciones, mayor carga de revisión

Las dos cosas siempre van juntas.

## Una configuración inicial sensata

### Local

```toml
model = "gpt-5.4"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
```

Y luego mantén narrow los writable roots.

### Cloud

- setup script determinista
- internet solo cuando haga falta
- exposición mínima de secretos
- comandos reales de verificación documentados en `AGENTS.md`

## Fallos típicos

### “Me pide approval todo el tiempo”
A menudo `read-only` es demasiado estricto, o el límite de escritura está mal definido.

### “Funciona local pero falla en cloud”
Suele haber estado local oculto que nunca se modeló en setup.

### “Ahora va más rápido, pero me da miedo”
Suele pasar cuando se amplían permisos sin mejorar review ni rollback.

## Regla práctica

Empieza con límites más estrechos.
Ábrelos solo cuando la fatiga de approvals sea el verdadero cuello de botella.
Y cada vez que amplíes permisos, refuerza review y disciplina de rollback.

El mejor sandbox de Codex no es el más poderoso.
Es el que tu equipo puede confiar una y otra vez.
