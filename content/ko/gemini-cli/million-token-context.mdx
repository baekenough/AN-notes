## 100만 토큰이 중요한 건 맞다. 하지만 전략이 더 중요하다

공식 저장소와 문서는 Gemini CLI의 핵심 장점으로 **1M token context**를 계속 강조합니다. 실제로 작은 도구들이 부분 샘플링에 그치는 상황에서, Gemini CLI는 저장소 단위의 넓은 추론이 가능합니다.

하지만 실전 교훈은 단순히 "매번 전체 저장소를 다 넣어라"가 아닙니다.

좋은 방식은 보통 이렇습니다.
- 먼저 **넓게 한 번 본다**
- 그다음 **위험한 파일/영역만 좁혀서 본다**
- 반복되는 규칙은 `GEMINI.md`로 빼서 프롬프트를 아낀다

## 컨텍스트 효율성: 100만 토큰이어도 아껴야 하는 이유

컨텍스트 윈도우가 크다고 해서 모든 파일을 통째로 읽어들이는(`read_file`) 것은 응답 속도 지연(Latency)과 불필요한 비용 증가를 초래합니다. Gemini CLI를 프로처럼 쓰려면 **외과 수술 같은 타겟팅(Surgical reads)**이 필요합니다.

1. **병렬 검색 활용**: `grep_search` 도구에 보수적인 제한(`total_max_matches`)을 걸고, 스코프(`include_pattern`)를 좁혀 병렬로 검색하게 하세요.
2. **충분한 컨텍스트와 함께 검색**: 정규식 검색 시 `context`, `before`, `after` 파라미터를 활용해 주변 코드를 함께 읽어오면, 파일을 추가로 열어보는 턴(Turn)을 아낄 수 있습니다.
3. **특정 라인만 읽기**: 큰 파일의 경우 전체를 읽지 말고 `start_line`과 `end_line`을 활용해 필요한 구간만 가져오세요.

## 실제로 좋은 패턴

### 1) 수정 전에 아키텍처 먼저 그리기

```bash
gemini "@src @content @package.json 이 앱의 아키텍처, 콘텐츠 파이프라인, 유지보수 리스크를 정리해줘"
```

클린업이나 마이그레이션 전에는 전역 지도가 먼저 필요합니다.

### 2) 서브 에이전트(Sub-agents) 위임: Codebase Investigator

저장소가 너무 방대해서 직접 하나하나 검색하기 까다로울 때는 특수 서브 에이전트인 `codebase_investigator` 도구를 활용하는 것이 훨씬 유리합니다.
"모호한 버그의 근본 원인을 찾아줘" 또는 "인증 시스템 전반의 아키텍처 리포트를 작성해줘"라고 요청하면, 해당 서브 에이전트가 스스로 파일을 탐색하고 메인 세션의 컨텍스트를 아끼면서 완벽한 요약본을 가져옵니다.

### 3) 여러 레이어를 건드리는 변경 영향 분석

```bash
gemini "@src @package.json 콘텐츠 모델을 바꾸려 한다. loader, route, UI, localization 중 어디가 영향받는지 영역별로 정리해줘"
```

Gemini CLI가 특히 좋은 지점이 여기입니다. 콘텐츠, UI, 설정, 문서를 한 턴에서 같이 볼 수 있습니다.

## 컨텍스트를 낭비하지 않는 방법

### 막연한 요청보다 구조화된 포함이 낫다

나쁜 예:
```text
전부 읽고 중요한 걸 말해줘.
```

더 좋은 예:
```text
@src @content @messages
콘텐츠 아키텍처, 로케일 전략, slug parity가 깨질 수 있는 지점을 설명해줘.
```

### 토큰 캐싱은 비용 구조를 바꾼다

공식 문서에 따르면 **토큰 캐싱**은 **Gemini API 키** 또는 **Vertex AI** 인증에서 사용할 수 있습니다. 반대로 Google 계정 OAuth 로그인에서는 현재 캐싱이 지원되지 않습니다.

즉 큰 컨텍스트 워크플로를 자주 돌릴수록 아래 조합이 유리합니다.
- `GEMINI_API_KEY`
- 또는 Vertex AI 인증

긴 세션에서는 `/stats`로 사용량과 cached token 절감량을 확인할 수 있습니다.
