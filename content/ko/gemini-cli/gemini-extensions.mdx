## Extensions는 단순한 플러그인보다 넓다

현재 Gemini CLI 문서는 extensions를 다음 요소를 묶는 패키지로 설명합니다.

- prompts
- MCP servers
- custom commands
- hooks
- sub-agents
- **agent skills** (에이전트 스킬)

즉 "명령 하나 추가하는 플러그인" 정도로 생각하면 범위를 너무 좁게 보는 셈입니다.

## 실전에서는 레이어로 생각하면 편하다

Gemini CLI 커스터마이징은 보통 세 층으로 나누면 정리가 잘 됩니다.

### 1단계: `GEMINI.md`
지속적인 지시문과 팀 규칙을 두는 곳입니다. (코딩 스타일, 저장소 제약 조건 등)

### 2단계: MCP 서버
외부 시스템이나 특수 도구 접근이 필요할 때 씁니다. (DB, Jira, 내부 API 등)

### 3단계: Extensions & Skills
워크플로를 **묶어서 배포/공유**하고 싶을 때 씁니다. 반복되는 내부 명령, 프롬프트 팩, 그리고 전문가 워크플로인 **Skills**가 여기에 포함됩니다.

## 에이전트 스킬 (Agent Skills)의 강력함

Gemini CLI의 가장 큰 차별점 중 하나는 `activate_skill` 도구를 통해 **특화된 전문가 모드**를 켤 수 있다는 점입니다.

- **기본 탑재 스킬들**: `security-review` (보안 취약점 리뷰), `code-review` (코드 리뷰), `web-clone` (URL 기반 UI 복제) 등의 특화된 지침을 불러와 에이전트의 행동 방식을 완전히 전문가 모드로 전환할 수 있습니다.
- **Skill-Creator**: 사용자는 `skill-creator` 스킬을 활성화하여 사내만의 독자적인 업무 프로세스(예: 사내 배포 규칙 적용기)를 새로운 스킬로 손쉽게 만들고 팀과 공유할 수 있습니다.

## 지금 바로 관리할 수 있는 것들

대화형 CLI 안에서는:
```text
/extensions list
```

터미널에서는:
```bash
gemini extensions list
# GitHub에서 설치
gemini extensions install https://github.com/gemini-cli-extensions/workspace
```

## 좋은 팀 도입 순서

1. `GEMINI.md`에 저장소 규칙 정리
2. 꼭 필요한 MCP 1~2개 연결
3. `skill-creator`로 사내 반복 작업을 스킬로 정의
4. 워크플로가 안정화되면 전체를 Extension으로 패키징하여 팀에 배포

## 이 프로젝트에 대한 추천

AN-Con이 더 커진다면 Gemini CLI 패키지는 아마 이런 구성이 좋습니다.
- 콘텐츠 parity 점검 커스텀 명령
- 언어별 번역 및 톤앤매너 리뷰 스킬 (`locale-reviewer` 스킬 생성)
- 추후 docs/CMS 연동용 MCP 포함

처음부터 확장을 만들지 말고, **`GEMINI.md` + 자체 생성 스킬**로 반복 패턴을 검증한 뒤 패키징하는 편이 낫습니다.
