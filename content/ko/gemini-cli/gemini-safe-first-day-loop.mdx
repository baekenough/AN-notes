## 속도보다 안전 루프를 먼저 만들기

Gemini CLI 입문 실수 대부분은 범위 과대와 trust boundary 미정의에서 시작됩니다.

첫날 목표:

- 안전한 설정
- 작은 성공 1개
- 검증 근거 확보

## 45분 첫날 루프

### 0~10분: 신뢰/권한 경계 확인

- trusted folder 확인
- sandbox mode 확인
- 수정 금지 범위 명시

### 10~25분: 좁은 작업 실행

작은 표면(소수 파일) 작업으로 시작합니다.

### 25~35분: 검증

최소 체크 실행 후 diff 범위 확인.

### 35~45분: 기록

- 변경 요약
- 실행 커맨드
- 남은 리스크

## 입문 프롬프트 형태

```text
Goal:

Context:

Constraints:

Done when:
```

`Done when`은 필수입니다.

## 입문 안티패턴

### 처음부터 대형 리팩토링 진입

루프를 익히기 전에 통제력을 잃습니다.

### trust 설정 불명확 상태 실행

코드 실수보다 경계 실수가 먼저 납니다.

### 근거 없는 완료 선언

확신 문장은 검증이 아닙니다.

## 빠른 체크리스트

- trust/sandbox 확인 완료
- 좁은 작업 1개 완료
- 검증 커맨드 실행
- 완료 메모 작성

이 4개면 첫 Gemini 루프는 건강합니다.
