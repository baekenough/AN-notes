> **공식 레퍼런스**: [Trusted folders](https://geminicli.com/docs/cli/trusted-folders/) · [Sandboxing](https://geminicli.com/docs/cli/sandbox/) · [Checkpointing](https://geminicli.com/docs/cli/checkpointing/)

## Gemini CLI에도 이제 꽤 분명한 안전 모델이 있다

AI 에이전트가 내 코드를 엉망으로 만들까 봐 걱정된다면, 아래의 4가지 다중 방어선을 이해하는 것이 중요합니다.

1. **Trusted Folders**: 워크스페이스에 대한 기본 기능 권한 제어
2. **Plan Mode**: 코드를 수정하기 전, 읽기 전용 상태로 설계 검증
3. **Sandboxing**: 위험한 셸 실행을 호스트 밖으로 격리
4. **Checkpointing**: AI가 파일을 쓰기 전에 되돌릴 지점을 생성

## Plan Mode: 실행 전 안전한 설계 (Read-Only)

규모가 큰 리팩터링이나 낯선 저장소 파악 시 가장 유용한 보안 기능은 **Plan Mode**(`/plan` 또는 `enter_plan_mode`)입니다.

- 에이전트가 코드를 탐색하고, 설계 문서를 작성하며, 전략을 짜는 동안 **어떤 파일도 수정하거나 파괴적인 셸 스크립트를 실행할 수 없도록** 강제합니다.
- 복잡한 요구사항일수록 이 모드에서 시작하세요. 완전히 검증된 계획이 나왔을 때만 수동으로 기본 모드로 돌아와 안전하게 변경을 적용(Execution)할 수 있습니다.

## Trusted folders: 낯선 저장소에서 자신을 보호하기

trusted folders는 **기본값으로 비활성화**되어 있습니다. 켜면 Gemini CLI가 프로젝트별 설정을 읽기 전에 해당 폴더를 신뢰할지 묻습니다.

사용자 설정에 추가:
```json
{
  "security": {
    "folderTrust": {
      "enabled": true
    }
  }
}
```

신뢰하지 않은(Untrusted) 상태에서는 `.env` 자동 로딩, 확장 설치, MCP 서버 연결 등 민감한 동작이 원천 차단됩니다. 외부 데모나 고객사 코드를 다룰 때 필수입니다.

## Sandboxing: 위험한 실행을 격리하기

Gemini CLI는 파일 수정이나 셸 실행을 호스트가 아니라 격리된 컨테이너 환경(Sandbox) 안에서 돌릴 수 있습니다.

```bash
gemini --sandbox
```

프로젝트별 추가 의존성이 필요하면 `.gemini/sandbox.Dockerfile`을 두어 맞춤형 격리 환경을 구성할 수 있습니다. 호스트 OS를 깨끗하게 유지하는 최고의 방법입니다.

## Checkpointing: AI가 파일 쓰기 전에 복구 지점 만들기

편집 범위가 큰 세션에서는 `checkpointing` 기능을 켜두는 것이 좋습니다. (기본값 꺼짐)
설정의 `general.checkpointing.enabled: true`를 적용하면, 파일 수정 직전에 섀도우 Git 스냅샷과 대화 기록이 저장됩니다.

만약 에이전트가 코드를 망쳤다면 즉시 복원할 수 있습니다.
```text
/restore
```

## 실제 사용 패턴 제안

- **익숙하지 않은 오픈소스/고객 저장소**: Trusted folders 글로벌 활성화 -> 신뢰 보류 -> **Plan Mode**로 구조 파악 -> **Sandboxing**으로 코드 실행 테스트
- **내가 관리하는 신뢰 저장소**: 폴더 Trust -> 일반 Interactive 모드 + 대형 수정 시에만 **Checkpointing** 켜기
