## 핵심 인사이트

Claude Code는 챗봇이 아닙니다 — 완벽한 기억력과 무한한 인내심을 가진 주니어 엔지니어입니다. 최고의 프롬프트는 유능한 팀원에게 주는 것과 같은 지시입니다.

## 프롬프팅 레벨

### Level 1: 직접 명령

간단하고 잘 정의된 작업에 적합:

```
src/auth/login.ts의 TypeScript 에러 수정해줘
```

```
회원가입 폼에 입력 유효성 검사 추가해줘 — 이메일 형식과 비밀번호 최소 8자
```

### Level 2: 컨텍스트 + 의도

판단이 필요한 작업에 더 적합:

```
/api/users 엔드포인트가 느려 (~2초). 프로파일링하고 최적화 방안 제안해줘.
데이터베이스 스키마는 변경하지 마 — 마이그레이션 중이야.
```

제약 조건("스키마 변경 금지")이 Claude가 잘못된 길로 가는 것을 방지합니다.

### Level 3: 아키텍처 위임

복잡한 다단계 작업에 적합:

```
결제 모듈을 콜백에서 async/await으로 리팩토링해줘.
1. stripe 연동부터 시작 (src/payments/stripe.ts)
2. 의존하는 주문 처리기 업데이트
3. 기존 테스트가 여전히 통과하는지 확인
4. PayPal 연동은 건드리지 마 — 다음 스프린트
```

번호가 매겨진 단계가 Claude에게 로드맵을 제공합니다. 제외 사항이 범위 확장을 방지합니다.

## 안티 패턴

### 모호한 요청

```
❌ "코드 좋게 만들어줘"
✅ "processOrder()의 순환 복잡도를 줄여줘 — 12개 분기가 있어"
```

### 한꺼번에 너무 많은 요청

```
❌ "전체 인증 시스템을 재작성하고, OAuth 추가하고, RBAC 구현하고, 2FA 추가해줘"
✅ "기존 인증 시스템에 Google OAuth 추가해줘. 현재 세션 관리는 유지해."
   (그 다음 RBAC와 2FA를 별도 작업으로)
```

### 컨텍스트 가정

```
❌ "버그 고쳐줘"
✅ "useCart의 경쟁 조건 수정해줘 — 아이템을 빠르게 추가하면 중복 항목이 생겨"
```

## "계획 먼저" 패턴

비정형 작업에는 Claude에게 먼저 계획을 세우게 하세요:

```
WebSocket으로 실시간 알림 추가하고 싶어.
코드 작성 전에 접근 방식을 정리해줘:
- 어떤 라이브러리 사용?
- WebSocket 서버는 어디에?
- 기존 인증과 어떻게 통합?
```

이렇게 하면 Claude가 잘못된 방향으로 500줄을 쓰기 전에 아키텍처 불일치를 잡을 수 있습니다.

## 확장 사고 모드

Claude Code는 작업의 복잡도에 따라 다른 깊이의 추론을 유도할 수 있어요:

| 상황 | 표현 예시 |
|------|----------|
| 일반 작업 | 기본 모드로 충분 |
| 복잡한 문제 | "깊이 생각해봐", "think hard" |
| 아키텍처 결정 | "ultrathink", "최대한 깊이 분석해줘" |

```
❌ "인증 구조를 어떻게 바꿀지 알려줘"
✅ "현재 세션 기반 인증을 JWT로 마이그레이션하는 방법을 ultrathink로 분석해줘.
   보안 trade-off, 롤백 전략, 단계별 전환 계획을 포함해서."
```

복잡한 리팩토링이나 중요한 설계 결정 전에 이 패턴을 쓰면 Claude가 엣지 케이스와 리스크를 훨씬 더 꼼꼼하게 짚어줘요.

## 대화 관리 테크닉

### 새 대화 시작 타이밍

작업 방향이 바뀌거나 컨텍스트가 무거워졌을 때는 새 대화를 시작하는 게 더 효율적이에요:

- 현재 작업이 끝나고 **다른 기능**으로 넘어갈 때
- Claude의 응답이 **이전 대화의 영향**을 과하게 받고 있을 때
- 대화가 길어져서 응답 품질이 떨어지는 느낌이 들 때

### /compact 활용

긴 대화 중 컨텍스트를 압축해서 토큰을 절약하고 집중력을 유지할 수 있어요:

```
/compact
```

`/compact`를 실행하면 Claude가 현재까지의 대화를 요약하고, 핵심 맥락만 유지한 채로 계속 작업할 수 있어요. 토큰 한도에 가까워질 때 특히 유용해요.

### 이미지/스크린샷 활용

UI 버그나 레이아웃 문제를 설명할 때, 스크린샷을 직접 붙여넣으면 훨씬 빠르게 문제를 전달할 수 있어요:

```
[스크린샷 붙여넣기]
"이 화면에서 모바일 뷰일 때 버튼이 겹쳐 보여. 수정해줘."
```

천 마디 설명보다 하나의 스크린샷이 더 정확한 컨텍스트를 제공해요.

## 반복 개선 패턴

Claude Code의 진짜 강점은 대화형 반복이에요. 한 번에 완벽한 프롬프트를 쓰려고 하지 말고, 단계별로 다듬어 가세요:

```
1차: "로그인 기능 추가해줘"
→ 결과 확인

2차: "좋아, 근데 에러 메시지를 사용자 친화적으로 바꿔줘.
     'Invalid credentials' 대신 '이메일 또는 비밀번호를 확인해 주세요' 형태로"
→ 결과 확인

3차: "마지막으로 rate limiting 추가해줘 — 5회 실패 시 5분 잠금"
```

각 단계가 명확하고 검증 가능한 단위일수록 전체 품질이 높아져요.

## 슬래시 커맨드 활용

Claude Code에는 작업 효율을 높이는 내장 커맨드가 있어요:

| 커맨드 | 기능 |
|--------|------|
| `/help` | 사용 가능한 커맨드와 기능 목록 보기 |
| `/compact` | 긴 대화의 컨텍스트를 압축 (토큰 절약) |
| `/clear` | 대화 초기화 — 새 작업 시작 시 유용 |
| `/cost` | 현재 세션의 API 비용 확인 |

CLAUDE.md에 커스텀 슬래시 커맨드를 등록해 프로젝트별 워크플로우를 자동화할 수도 있어요.

## "보여줘 vs 해줘" 패턴

설명만 요청하는 것과 행동을 유도하는 것은 결과가 크게 달라요:

```
❌ "인증 시스템이 어떻게 동작해?"
   → Claude가 설명만 해줌

✅ "인증 시스템을 분석하고 보안 취약점이 있으면 수정해줘"
   → Claude가 실제로 코드를 보고 고쳐줌
```

```
❌ "이 API가 느린 이유가 뭐야?"
   → 가능한 원인 목록을 받음

✅ "이 API가 느린 이유를 프로파일링해서 찾아내고, 수정 가능한 부분은 바로 고쳐줘"
   → 진단 + 수정까지
```

Claude Code는 분석가가 아니라 엔지니어예요. 행동을 요청하세요.

## 파일 참조로 정확한 컨텍스트 주기

프롬프트에서 파일을 직접 언급하면 Claude가 해당 파일을 읽어요. 전체 코드베이스를 검색하게 하는 대신, 관련 파일을 직접 알려주는 게 훨씬 효율적이에요:

```
src/types/user.ts를 먼저 읽고, avatar_url 필드를 추가해줘
```

```
src/auth/login.ts를 봐봐 — 토큰 갱신 로직이 이상한 것 같아
```

전체 경로든 상대 경로든 상관없어요. 파일을 프롬프트 입력 창에 드래그 앤 드롭해서 빠르게 컨텍스트를 제공할 수도 있어요.

**핵심 원칙**: Claude가 코드베이스 전체를 탐색하길 기대하기보다, 관련 파일을 직접 가리켜 주세요. 더 빠르고, 더 정확한 결과를 얻을 수 있어요.

## 비대화형 모드 활용

`claude --print` (또는 `claude -p`)를 쓰면 대화 없이 한 번에 결과를 받을 수 있어요. 스크립트나 파이프라인에 Claude를 통합할 때 유용해요:

```bash
# 에러 로그 요약
cat error.log | claude -p "이 에러들을 요약해줘"

# 결과를 파일로 저장
claude -p "src/utils.ts에 대한 테스트를 생성해줘" > tests/utils.test.ts

# JSON 형식으로 출력
claude -p --output-format json "src/api/의 모든 export 함수 목록을 만들어줘"
```

### 실용적인 예시

**Git 커밋 메시지 자동 생성:**
```bash
git diff --staged | claude -p "write a conventional commit message"
```

**CI에서 코드 리뷰:**
```bash
claude -p "review this diff for security issues" < pr.diff
```

**문서 자동 생성:**
```bash
claude -p "generate JSDoc for all exported functions in src/api/"
```

Makefile, 셸 스크립트, git hook 등 어디서든 활용할 수 있어요.

## 모델 선택 전략

`/model` 커맨드로 대화 중에 모델을 전환할 수 있어요. 작업의 복잡도에 맞는 모델을 선택하면 비용과 성능을 최적화할 수 있어요:

| 작업 | 권장 모델 |
|------|----------|
| 아키텍처 리뷰, 복잡한 디버깅 | opus |
| 기능 구현, 코드 리뷰 | sonnet (기본값) |
| 간단한 질문, 파일 검색, 포맷팅 | haiku |

비대화형 모드에서는 `--model` 플래그를 사용해요:
```bash
claude -p --model haiku "이 에러 메시지 설명해줘"
```

**비용 참고**: haiku는 opus 대비 토큰당 약 10배 저렴해요. sonnet으로 시작해서, 복잡한 추론이 필요하면 opus로 올리고, 대량 작업에는 haiku를 사용하는 게 최적의 전략이에요.

## Permission 모드 활용

`--allowedTools` 플래그로 특정 도구를 미리 승인하면 권한 프롬프트를 줄일 수 있어요:

```bash
claude --allowedTools "Edit,Write,Bash(git *)"
```

`.claude/settings.json`에 영구적으로 설정할 수도 있어요:

```json
{
  "permissions": {
    "allow": ["Edit", "Write", "Bash(npm test)", "Bash(npm run lint)"]
  }
}
```

완전 자동화된 스크립트(CI 환경 전용)에서는 `--dangerouslySkipPermissions`를 사용할 수 있지만, 대화형 세션에서는 절대 사용하지 마세요.

**원칙**: 처음에는 제한적으로 시작하고, 신뢰할 수 있는 도구를 점진적으로 허용해 나가세요.

## Codex 비교

GPT Codex는 다르게 작동합니다 — 작업 설명을 미리 작성하면 자율적으로 실행합니다. 대화형 반복이 없습니다. 즉:
- Codex 프롬프트는 처음부터 더 완전해야 함
- Claude Code는 반복적 개선이 가능
- 둘 다 명시적 제약과 예시가 효과적
