## 왜 측정은 복원력 통제인가

신뢰성은 눈에 띄는 장애 전에 서서히 악화됩니다.
지표는 그 드리프트를 조기에 알려주는 경보 체계입니다.

## 핵심 복원력 지표 세트

| 지표 | 의사결정 질문 | 대표 소스 |
|---|---|---|
| MTTD | 탐지가 충분히 빠른가? | 알림 타임라인 |
| MTTC | 의사결정이 지연되는가? | 사고 커맨드 로그 |
| MTTR | 안정 상태를 얼마나 빨리 복구하는가? | 배포/검증 타임라인 |
| Evidence freshness | 종료 근거가 최신인가? | 검증 산출물 로그 |
| Follow-up closure rate | 하드닝 약속이 실제로 닫히는가? | 사고 후 백로그 |

## 심각도 기반 SLO 설계

- **SEV-1**: 엄격한 격리 및 롤백 활성화 시간 창
- **SEV-2**: 주요 저하 안정화 시간 창
- **SEV-3**: 계획된 보정 릴리스 완료 시간 창

SLO는 숫자로 정의하고 오너를 지정해야 합니다.

## 측정 프로토콜

모든 사고마다 아래 시각을 기록하세요.

- 사고 시작
- 최초 탐지
- 최초 완화 의사결정
- 안정 상태 확인
- 종료

타임스탬프 체인이 비면 분석 신뢰도가 떨어집니다.

## 주간 신뢰성 운영 리뷰

1. 임계값 위반 점검
2. tail-latency 이상치 확인
3. 실패를 통제 포트폴리오 버킷에 매핑
4. 영향도가 큰 회귀 항목 오너/기한 지정

## 임계값 에스컬레이션 모델

지표별 green/yellow/red 경계를 정의합니다.
red 도달 시 필수 항목:

- 즉시 에스컬레이션 기록
- 명시적 reliability owner 지정
- 다음 사이클 강제 재검증

## 대시보드 품질 규칙

- 추세와 분위수 대역 표시
- 심각도 클래스 분리
- 분모와 표본 수 포함
- 모든 스파이크를 사고 노트와 연결

## 분기별 보정 루프

분기마다:

- 안정 달성 후에만 SLO 상향
- 행동으로 이어지지 않는 지표 폐기
- 신규 실패 클래스용 지표 1개 추가

목표는 대시보드 크기가 아니라 의사결정 품질입니다.

## 고급 안티패턴

### tail-risk를 평균으로 덮어버리기

평균값은 사고를 만드는 이벤트를 숨깁니다.

### 책임 오너 없는 SLO

오너 없는 SLO는 상태 보고용 수치가 됩니다.

### 오래된 근거로 종료 선언

오래된 증거는 현재 상태 신뢰도를 보장하지 못합니다.

## 빠른 체크리스트

월간 복원력 리뷰 전:

- 지표 정의 최신화
- 심각도별 SLO 공개 + 오너 지정
- red/yellow 위반 항목 명시 배정
- 후속 종료율 추세 점검

Codex는 실행 속도를 높입니다. 지표는 신뢰성 무결성을 지켜줍니다.

## 지표 사전 템플릿 (필수 필드)

각 지표를 동일 스키마로 정의하세요.

```md
### Metric Definition
- Name:
- Purpose:
- Formula:
- Data source:
- Collection cadence:
- Owner:
- Red threshold:
- Yellow threshold:
- Expected action on breach:
```

정의가 모호하면 사고 중 해석 논쟁이 반복됩니다.

## Error-budget형 SLO 정책

심각도 클래스마다 운영 예산을 설정합니다.

- 분기당 허용 breach 횟수
- 의무 에스컬레이션 임계값
- 예산 소진 시 freeze 규칙

### 예시 정책

- SEV-1: containment window 미달 0회 허용
- SEV-2: 분기 2회 breach 초과 시 통제 리뷰 필수
- SEV-3: 즉시 동결 대신 추세 관리

## 주간 리뷰 질문 세트

1. 기준선 대비 가장 크게 움직인 지표는?
2. 샘플 수 기준으로 signal인가 noise인가?
3. 이번 주 action owner는 누구인가?
4. 어느 통제 포트폴리오 버킷에 반영할 것인가?
5. 다음 리뷰까지 기대되는 관측 변화는?

## 에스컬레이션 매핑 테이블

| breach 유형 | 1차 오너 | 2차 오너 | 응답 SLA |
|---|---|---|---|
| MTTD red | observability owner | incident commander | 24h |
| MTTC red | incident commander | release owner | same day |
| MTTR red | platform owner | service owner | 24h |
| freshness breach | verifier owner | commander | same day |
| follow-up closure breach | reliability owner | team lead | 72h |

## 월간 요약 템플릿

```md
### Monthly Resilience Summary

- Top regressing metric:
- Repeated breach classes:
- Controls added this month:
- Controls retired this month:
- Ownership risks:
- Next-month focus:
```

짧고 의사결정 중심으로 유지하세요.

## 데이터 품질 체크

대시보드를 신뢰하기 전 아래를 확인합니다.

- 누락 타임스탬프 비율
- 중복 incident ID
- 심각도 라벨 불일치
- 데이터 소스 갱신 지연

잘못된 데이터 위의 정밀 지표는 여전히 오해를 만듭니다.

## 안티-게이밍 규칙

- 단일 지표 순위로 팀 평가 금지
- 지표 개선 주장 시 근거 링크 의무화
- 평균 개선 전 tail percentile 우선 확인
- 보상은 1주 스파이크가 아니라 지속 추세 기준

이 규칙이 조직 압박 속에서도 지표 무결성을 지킵니다.

<!-- expansion:depth-v3 -->
## 지표 리뷰 보드 운영 규칙

월간 신뢰성 보드 출력은 3개만 허용합니다.

1. **keep** — 여전히 액션을 유도함
2. **change** — 정의/임계값 수정 필요
3. **remove** — 의사결정 가치 없음

이 규칙이 대시보드 비대화를 막습니다.

## tail-risk 추적

평균 외에 반드시 추적:

- MTTD/MTTR의 p90/p95/p99
- 가장 오래 열린 follow-up 나이
- 최악 심각도 breach 재발 간격

tail 지표가 진짜 사고 리스크를 보여줍니다.

## SLO breach 플레이북

breach 발생 시:

- same day breach 기록 생성
- owner + verifier 지정
- 교정 통제 후보 정의
- 7일 내 리뷰 체크포인트 지정

통제 효과 근거 없이는 breach 기록을 닫지 않습니다.

## 지표 폐기 기준

아래를 모두 만족할 때 폐기:

- 2분기 동안 액션 유도 없음
- 다른 지표와 중복 강함
- 이해관계자가 활용 방식을 설명 못함

조용히 삭제하지 말고 폐기 노트를 남기세요.
