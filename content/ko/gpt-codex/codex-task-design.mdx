## 핵심 원칙: 추측할 여지를 줄여라

Codex는 강한 실행기이지만, 사용자가 모호하게 말할수록 모델이 추측해야 할 영역이 커집니다.
좋은 프롬프트의 목적은 “멋진 문장”이 아니라 **추측 비용을 줄이는 것**입니다.

## 가장 잘 먹히는 기본 틀

공식 best practices는 이슈나 PR 설명처럼 쓰는 방식을 권장합니다.

```text
Goal:
무엇을 바꾸고 싶은가

Context:
어떤 파일, 에러, 기존 패턴이 중요한가

Constraints:
무엇을 건드리면 안 되는가

Done when:
무엇이 완료 기준인가
```

이 구조는 단순하지만 강력합니다.
Goal만 있고 Done when이 없으면 끝을 알 수 없고, Context가 없으면 저장소 내부 패턴을 추측하게 됩니다.

## 예시: 나쁜 프롬프트 vs 좋은 프롬프트

나쁨:

```text
cart 쪽 개선해줘
```

나음:

```text
Goal:
cart drawer의 수량 변경에 optimistic update를 추가해.

Context:
- UI는 src/components/cart/ 아래에 있음
- server action은 src/app/actions/cart.ts 에 있음
- loading 상태 패턴은 wishlist 구현을 따를 것

Constraints:
- public API shape는 바꾸지 말 것
- 새 상태관리 라이브러리 추가 금지

Done when:
- 수량 변경이 즉시 반영되어 보일 것
- 실패 시 rollback 될 것
- lint, build, 관련 테스트가 통과할 것
```

## scope는 검증 루프 하나로 잘라라

작업 크기를 잡을 때 가장 좋은 기준은 “기능 단위”가 아니라 **검증 루프 단위**입니다.

좋은 작업:

- 한 폴더 안의 refactor
- 재현 가능한 버그 수정
- 특정 API 변경에 맞춘 문서 업데이트

나쁜 작업:

- auth, onboarding, billing을 한 번에 재설계
- 기술부채를 전체적으로 정리
- 앱을 전반적으로 modernize

한 번의 lint/build/test 루프로 건강을 판단하기 어려우면 대개 너무 큽니다.

## 계획부터 세워야 하는 작업도 있다

다음 조건이면 먼저 plan을 요청하는 편이 낫습니다.

- 파일/도메인이 세 군데 이상 걸침
- 구현보다 설계 판단 비중이 큼
- rollback 비용이 큼
- 여러 reasonable solution이 있음

이럴 때는 바로 구현보다 아래처럼 시작하세요.

```text
먼저 구현하지 말고 plan만 제안해.
위험, 변경 파일, 검증 방법을 포함해.
```

## 로컬 예시를 주면 추측이 줄어든다

Codex는 추상적인 “우리 스타일대로”보다 구체적인 “이 파일처럼”에 훨씬 강합니다.

좋은 예:

- `src/features/profile/actions.ts` 패턴을 따르기
- `docs/releases/` 포맷에 맞추기
- `Button` 컴포넌트 variant naming을 재사용하기

가능한 한 **기존 구현을 참조점으로 붙여주세요.**

## 제약 조건은 결과 품질을 올린다

제약은 Codex를 묶는 족쇄가 아니라, 팀 기준으로 유도하는 가드레일입니다.

특히 자주 쓰는 제약:

- 새 의존성 추가 금지
- public API shape 유지
- 특정 디렉터리 밖 수정 금지
- generated file 수정 금지
- diff를 작게 유지

## 언제 skill, MCP, subagent로 승격할까

같은 작업을 여러 번 반복하면 프롬프트만 계속 다듬지 말고 표면을 바꾸는 게 낫습니다.

- **반복 작업**이면 → [Skills](./codex-skills)
- **repo 밖 컨텍스트**가 필요하면 → [MCP](./codex-mcp)
- **병렬 탐색/구현**이 유리하면 → subagent 요청 고려

공식 subagents 문서는 Codex가 **명시적으로 요청했을 때만** subagent를 띄우고, 현재 sandbox policy를 상속한다고 설명합니다. 즉, 복잡한 작업을 병렬화할 수 있지만 비용도 같이 늘어납니다.

## 불확실하면 Best-of-N

OpenAI는 어려운 작업에서 Best-of-N을 강조합니다.
특히 아래에서 유용합니다.

- 네이밍과 구조 선택이 중요한 작업
- 구현 후보가 둘 이상 타당한 작업
- 한 번에 merge하기 전 비교 검토가 필요한 작업

## 빠른 출발 체크리스트

프롬프트를 보내기 전에 아래 다섯 가지만 확인해도 품질이 달라집니다.

- 목표가 한 문장으로 말해지는가?
- 관련 파일/폴더를 적었는가?
- 금지할 것을 적었는가?
- 완료 기준을 측정 가능하게 적었는가?
- 한 번의 review로 판단 가능한 크기인가?

Codex 작업 설계의 본질은 특별한 프롬프트 공식이 아니라,
**사람 엔지니어에게도 명확한 작업을 만드는 것**입니다.
