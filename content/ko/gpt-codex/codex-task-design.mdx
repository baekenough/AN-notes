## 핵심 원칙

Codex는 작업이 실제 엔지니어링 티켓처럼 생겼을 때 가장 잘 동작합니다.

OpenAI도 Codex 프롬프트를 GitHub 이슈나 PR 설명처럼 구조화하라고 권장합니다. 실무적으로는 다음 네 가지가 중요합니다.

- 명확한 목표
- 파일/모듈 단위의 구체적 컨텍스트
- 나쁜 해법을 막는 제약
- 분명한 완료 조건

“이거 좀 개선해줘” 같은 말만 던지면, 실제 과업을 모델이 스스로 발명해야 합니다.

## 이슈 형태 프롬프트 템플릿

중간 이상 규모의 작업은 이런 구조가 좋습니다.

```text
Goal:
cart drawer에 optimistic update를 추가해.

Context:
- UI는 src/components/cart/ 에 있음
- server action은 src/app/actions/cart.ts 에 있음
- src/components/wishlist/ 의 패턴을 따를 것

Constraints:
- API shape는 바꾸지 말 것
- 새로운 상태 관리 라이브러리 추가 금지
- diff는 작고 국소적으로 유지할 것

Done when:
- 변경이 즉시 반영되어 보일 것
- 실패 시 rollback 될 것
- lint, typecheck, cart 테스트가 통과할 것
```

이 형식은 사람에게도, Codex에게도 리뷰가 쉽습니다.

## 스코프가 전부다

Codex의 강점은 경계가 있는 실행입니다. 그래서 중요한 건 “완벽한 프롬프트 문장”보다 **작업을 알맞은 크기로 자르는 것**입니다.

좋은 작업 형태:

- 한 서브시스템의 한 기능
- 한 영역에 국한된 리팩토링
- 재현 가능한 실패가 있는 버그
- 여러 파일에 반복 적용되는 마이그레이션 패턴

좋지 않은 작업 형태:

- “앱 전체를 현대화해줘”
- “모든 기술 부채를 해결해줘”
- “auth, onboarding, billing을 한 번에 갈아엎어줘”

사람 엔지니어에게 먼저 설계 문서를 요구할 만한 일이라면, Codex 작업도 대개 너무 큽니다.

## 언제 작업을 쪼개야 하나

다음과 같으면 여러 Codex 작업으로 나누는 편이 좋습니다.

- 파일이 서로 다른 도메인에 흩어져 있을 때
- 검증 방식이 서로 다를 때
- 한 부분은 탐색적이고 다른 부분은 결정적일 때
- 첫 결과를 보고 다음 단계를 승인하고 싶을 때

예시:

1. DB 필드와 스키마 wiring 추가
2. API 핸들러 업데이트
3. UI와 테스트 수정

한 방에 끝내는 것보다 덜 화려해 보여도, 훨씬 신뢰성이 높습니다.

## 추상 조언보다 저장소 예시를 줘라

Codex는 일반론보다 저장소 안의 구체적 앵커에 더 잘 반응합니다.

더 좋은 예:

```text
src/features/profile/actions.ts 와 같은 방식으로 구현해.
```

덜 좋은 예:

```text
우리 아키텍처에 맞게 구현해.
```

기존 패턴을 정확히 가리킬수록 Codex의 추측 비용이 줄어듭니다.

## 규칙은 AGENTS.md에, 의도는 프롬프트에

흔한 안티패턴은 영구 규칙을 모든 작업 프롬프트에 반복해서 넣는 것입니다.

더 나은 방식:

- 지속 규칙은 `AGENTS.md`
- 오늘의 과업은 프롬프트

예:

- `AGENTS.md`: lint + typecheck 실행, generated 파일 수정 금지
- 프롬프트: billing export에 draft invoice 지원 추가

## 제약은 결과를 더 좋게 만든다

제약은 방해가 아니라 방향입니다.

가치가 큰 제약 예시:

- 의존성 추가 금지
- 공개 API shape 변경 금지
- 특정 폴더 안에서만 수정
- 기존 테스트 보존
- 새 추상화 대신 기존 유틸 재사용

제약이 없으면 문제는 풀 수 있어도 팀이 원한 방식이 아닐 수 있습니다.

## “Done”을 관찰 가능한 형태로 적어라

결과 품질을 높이는 가장 쉬운 방법 중 하나는 완료 조건을 관찰 가능한 기준으로 적는 것입니다.

좋은 예:

- `pnpm lint` 통과
- `auth.spec.ts`의 실패 테스트 수정
- empty state UI가 기존 dashboard 카드 스타일과 일치
- `src/payments/` 밖 변경 없음

“보기 좋게” 같은 기준은 너무 약합니다.

## 불확실한 작업엔 Best-of-N

OpenAI는 난도가 있는 작업에서 **Best-of-N** 패턴을 강조합니다.

다음 경우에 유용합니다.

- 여러 구현안이 모두 그럴듯할 때
- 이름과 구조 선택이 중요할 때
- merge 전 tradeoff를 비교하고 싶을 때

## Task Queue를 가벼운 백로그처럼 써라

OpenAI는 Codex task queue를 lightweight backlog처럼 쓰는 것도 권장합니다.

예를 들면:

- 흐름을 끊지 않고 곁다리 수정 아이디어 저장
- 나중에 검토할 cleanup 작업 적재
- 큰 작업 뒤 follow-up 작업 예약

모든 Codex 작업이 바로 최종 PR이 될 필요는 없습니다.

## 빠른 체크리스트

비단순 작업을 보내기 전에 확인하세요.

- 목표가 하나인가?
- 관련 파일/디렉터리를 적었는가?
- 중요한 제약을 적었는가?
- 성공 검증 방법을 적었는가?
- 다른 엔지니어가 봐도 그럴듯한 이슈처럼 읽히는가?

그렇다면 작업 설계가 대체로 잘 된 것입니다.
