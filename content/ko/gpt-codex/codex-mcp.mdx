## 왜 MCP가 중요한가

많은 사용자가 외부 문서, 티켓, 대시보드 정보를 프롬프트에 복붙합니다.
하지만 이 방식은 금방 한계가 옵니다.

- 최신성이 떨어짐
- 길어짐
- 출처 확인이 어려움
- 반복 작업에 약함

MCP의 핵심 가치는 **repo 밖의 컨텍스트를 구조적으로 연결**한다는 데 있습니다.

## 어떤 경우에 특히 유용한가

- 내부 문서 포털 검색
- Jira, Linear, GitHub issue 조회
- 디자인/사양 시스템 참조
- 사내 API 문서, runbook, playbook 조회
- 데이터베이스나 운영 도구에 대한 제한적 조회

즉, Codex가 코드만 보고는 모를 맥락이 필요할 때 잘 맞습니다.

## 기본 설정 방법

MCP 문서에 따르면 Codex의 MCP 설정은 기본적으로 `~/.codex/config.toml`에 저장되며, **CLI와 IDE extension이 이 구성을 공유**합니다.
trusted project라면 `.codex/config.toml`로 프로젝트 단위 설정도 가능합니다.

CLI로는 이렇게 추가합니다.

```bash
codex mcp add context7 -- npx -y @upstash/context7-mcp
```

그리고 세션 안에서는 `/mcp`로 현재 연결된 MCP 도구를 확인할 수 있습니다.

## MCP를 붙인 뒤 프롬프트가 달라져야 한다

MCP를 연결했는데도 예전처럼 길게 복붙하면 이점이 줄어듭니다.
대신 아래처럼 요청하세요.

```text
이 repo의 결제 로직을 보기 전에,
연결된 issue tracker와 runbook에서 refund 관련 최근 변경 사항을 먼저 요약해줘.
그 다음 코드 영향 범위를 정리해줘.
```

핵심은 “외부 문서를 읽고 오라”가 아니라,
**어떤 외부 맥락을 어떤 순서로 써야 하는지 지정하는 것**입니다.

## 좋은 MCP 운영 원칙

### 1. read-only부터 시작
처음부터 write capability가 있는 서버를 넓게 붙이지 마세요.

### 2. 신뢰 가능한 서버만
문서, 티켓, 검색 같이 가치가 높은 시스템부터 붙이는 편이 좋습니다.

### 3. 팀 규칙은 AGENTS에 남기기
어떤 MCP 서버를 신뢰하는지, 어떤 용도로 쓰는지 `AGENTS.md`에 적어두면 좋습니다.

### 4. 시크릿과 권한을 최소화하기
MCP도 결국 외부 시스템 접점입니다. 샌드박스와 동일하게 최소 권한 원칙이 중요합니다.

## skill과 함께 쓰면 더 강하다

MCP는 컨텍스트를 가져오고, skill은 절차를 패키징합니다.
둘을 조합하면 반복적인 업무 흐름이 강해집니다.

예시:

- release note skill이 MCP로 이슈/PR 정보를 읽고 초안을 생성
- bug triage skill이 incident 문서와 ticket 상태를 함께 참조
- docs update skill이 공식 문서를 MCP로 조회해 drift를 점검

## 흔한 실패 패턴

### “서버는 연결했는데 Codex가 안 쓴다”
프롬프트에서 어떤 외부 맥락이 필요한지 분명히 말하지 않았을 수 있습니다.

### “연결이 너무 많아 무엇을 믿어야 할지 모르겠다”
팀이 신뢰하는 MCP 서버 집합을 작게 유지하는 편이 낫습니다.

### “MCP가 있으니 sandbox는 덜 중요하다”
오히려 반대입니다. 외부 연결이 늘수록 권한 설계는 더 중요해집니다.

## 빠른 체크리스트

- 정말 repo 밖 정보가 필요한가?
- 복붙보다 연결이 더 낫나?
- read-only로 시작할 수 있나?
- 어떤 서버를 신뢰하는지 팀이 합의했나?
- 프롬프트에서 MCP를 어떤 순서로 쓸지 명시했나?

Codex를 “코드 편집기”에서 “맥락을 연결한 작업자”로 바꾸는 전환점이 바로 MCP입니다.
