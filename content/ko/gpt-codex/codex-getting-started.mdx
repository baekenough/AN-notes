## 먼저 알아둘 것

Codex를 잘 시작하는 핵심은 “세팅을 다 이해한 뒤 시작”이 아닙니다.
**작은 로컬 작업을 안전하게 한 번 성공시키는 것**이 더 중요합니다.

초반에는 아래 네 가지만 잡으면 됩니다.

- 어떤 클라이언트에서 쓸지
- 어떤 저장소를 열지
- 첫 작업을 얼마나 작게 자를지
- 언제 되돌릴 수 있게 만들지

## 어디서 시작할까

공식 Quickstart 기준으로 Codex는 앱, CLI, IDE 확장, cloud 흐름으로 들어갈 수 있습니다.

### Codex 앱
가장 무난한 입문 경로입니다.
리뷰, worktree, 자동화 같은 고급 기능까지 한 UI에서 보기 쉽습니다.

### CLI
터미널 중심 개발자에게 좋습니다.
저장소를 바로 열고 명령어·diff·검증 루프를 붙이기 쉽습니다.

### IDE 확장
에디터 안에서 Codex를 붙이고 싶을 때 좋습니다.
다만 처음 감을 잡는 단계에서는 앱이나 CLI가 더 명확한 편입니다.

### Cloud
처음부터 cloud로 갈 필요는 없습니다.
로컬에서 한두 번 성공 패턴을 만든 다음 cloud 환경으로 옮기는 편이 덜 헷갈립니다.

## 첫 10분 흐름

공식 Quickstart는 다음 순서를 권장합니다.

1. Codex 앱 또는 CLI 설치
2. **ChatGPT 계정 또는 OpenAI API key로 로그인**
3. 작업할 프로젝트 폴더 선택
4. 첫 메시지 전송

문서상으로는 API key로도 로그인할 수 있지만, 일부 기능은 ChatGPT 계정 로그인 때 더 잘 맞습니다. 입문자라면 일단 **로컬 프로젝트 + 기본 계정 로그인** 조합이 가장 단순합니다.

## 처음엔 로컬로 시작하자

처음부터 cloud, automation, 넓은 권한 설정까지 동시에 배우면 어디서 꼬였는지 구분이 어려워집니다.

초반에는 아래 같은 작업이 좋습니다.

- “이 저장소 구조를 5줄로 설명해줘”
- “이 에러를 재현하는 파일부터 찾아줘”
- “이 폴더에서 dead code 후보를 찾아줘”
- “테스트를 깨지 않게 작은 리팩토링만 해줘”

즉, **탐색 가능하고 되돌리기 쉬운 작업**으로 시작하는 게 좋습니다.

## Git 체크포인트를 습관으로

Quickstart도 체크포인트를 권장합니다. 이유는 단순합니다. Codex는 빠르게 많이 바꿀 수 있기 때문입니다.

초반 추천 루프는 다음과 같습니다.

```bash
git status
git add -A && git commit -m "checkpoint: before codex task"
```

작업 후에도 마음에 들면 다시 checkpoint를 남기면 됩니다.
이렇게 해두면 “좋아 보이는데 어디가 바뀌었는지 모르겠다”는 상황을 줄일 수 있습니다.

## 처음 배울 핵심 명령 4개

Slash commands 문서를 기준으로 초반엔 이 네 개면 충분합니다.

### `/init`
현재 디렉터리에 `AGENTS.md` 초안을 만듭니다.
저장소 규칙을 반복 설명하기 싫어질 때 가장 먼저 익힐 명령입니다.

### `/review`
현재 워킹트리를 리뷰합니다.
작업 직후 한 번 더 보수적으로 검토할 때 좋습니다.

### `/mcp`
연결된 MCP 서버를 확인합니다.
“Codex가 지금 외부 도구를 볼 수 있는가?”를 점검할 때 유용합니다.

### `/status`
현재 모델, approval policy, writable roots 같은 세션 상태를 확인합니다.
이상하게 막힐 때 제일 먼저 보는 명령입니다.

## 첫 프롬프트는 소원보다 작업 지시

처음부터 “앱 전체를 현대화해줘” 같은 말은 피하세요.
대신 아래처럼 쓰는 편이 낫습니다.

```text
Goal:
src/lib/ 안에서 사용되지 않는 유틸 후보를 찾아줘.

Constraints:
- 파일 수정은 하지 말고 조사만
- 후보마다 근거가 되는 import/usage를 보여줄 것

Done when:
- 우선순위가 높은 후보 3개와 이유를 정리할 것
```

이 정도만 되어도 Codex는 훨씬 안정적으로 움직입니다.

## 첫 주 체크리스트

Codex 입문 첫 주에는 아래 순서가 좋습니다.

1. 로컬 프로젝트에서 작은 read-only 작업
2. 작은 수정 + lint/test 실행
3. `/review`로 diff 다시 보기
4. `/init`으로 저장소용 `AGENTS.md` 초안 만들기
5. 반복 작업이 생기면 skill 또는 automation으로 승격 여부 검토

처음부터 모든 기능을 다 쓰려 하기보다, **작업 → 검토 → 규칙화** 순서로 가면 오래 갑니다.

## 빠른 시작 요약

- 입문은 **앱 또는 CLI + 로컬 프로젝트**가 가장 쉽습니다.
- 첫 작업은 작고 되돌리기 쉬워야 합니다.
- Git 체크포인트를 초반부터 습관화하세요.
- `/init`, `/review`, `/mcp`, `/status`만 익혀도 생산성이 빨리 올라갑니다.
- 더 깊은 사용은 다음 문서들에서 `AGENTS.md`, sandbox, task design으로 이어가면 됩니다.
