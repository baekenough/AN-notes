## Codex에는 두 가지 실행 모델이 있다

Codex에서 “샌드박스”는 상황에 따라 의미가 조금 다릅니다.

### 로컬 Codex (CLI / IDE)

Codex가 **내 머신에서 실행**되지만, 사용자가 정한 권한 경계 안에서 동작합니다.

### Codex Cloud

Codex가 **OpenAI가 관리하는 컨테이너** 안에서 실행됩니다. 내 호스트 머신은 직접 노출되지 않고, 환경은 클라우드 설정과 setup script로 준비합니다.

이 둘을 구분하지 않으면 설정이 금방 헷갈리기 시작합니다.

## 로컬 Sandbox Mode

OpenAI 문서는 로컬에서 세 가지 주요 sandbox mode를 설명합니다.

| 모드 | Codex가 할 수 있는 일 | 적합한 용도 |
|------|-----------------------|-------------|
| `read-only` | 파일 조회만 가능, 수정/명령 실행은 승인 필요 | 감사, 코드 리뷰, 안전한 탐색 |
| `workspace-write` | 워크스페이스 내부 수정과 일반 명령 실행 가능 | 일상적인 개발 작업 |
| `danger-full-access` | 샌드박스 제한 없음 | 매우 신뢰된 자동화에만 제한적으로 사용 |

대부분의 개발자에게는 `workspace-write`가 가장 현실적인 기본값입니다. 승인 피로를 줄이면서도 머신 전체를 넘겨주지는 않습니다.

## Approval Policy

sandbox mode는 **기술적으로 무엇이 가능한지**를 정합니다.
approval policy는 **언제 멈추고 사용자에게 물어야 하는지**를 정합니다.

대표 정책은 다음과 같습니다.

- `untrusted` — 신뢰 목록 밖 명령 전 승인 요청
- `on-request` — 샌드박스 안에서는 진행하고, 경계를 넘을 때 요청
- `never` — 승인 프롬프트 없이 진행

OpenAI가 강조하는 비교적 안전한 로컬 자동화 조합은 사실상 다음과 같습니다.

```text
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

반대로 `danger-full-access`와 `never`를 같이 쓰는 것은 훨씬 더 위험합니다.

## 핵심 목적은 “무한 권한”이 아니라 “승인 피로 감소”다

샌드박스는 보안 장치이기도 하지만, 동시에 작업 흐름을 부드럽게 만드는 장치이기도 합니다.

적절한 경계를 주면 Codex는:
- 저장소를 읽고
- 수정해야 할 파일을 바꾸고
- lint, typecheck, test를 돌리고
- 정말 더 큰 권한이 필요할 때만 멈춥니다

이 상태가 되어야 자율 코딩이 실제로 쓸 만해집니다.

## 클라우드 태스크의 실행 흐름

OpenAI 문서 기준으로 Codex cloud task는 대체로 다음 순서로 실행됩니다.

1. 컨테이너를 만들고 저장소를 checkout
2. **setup script** 실행
3. 인터넷 접근 설정 적용
4. 에이전트가 코드 수정 및 검증 수행
5. diff와 최종 결과 반환

여기서 특히 중요한 두 가지는:

- **setup script는 인터넷이 켜진 상태에서 실행됨**
- **agent internet access는 기본적으로 꺼져 있음**

즉, 다운로드나 도구 설치, 서비스 부트스트랩이 필요하면 setup 단계에서 끝내야 합니다.

## setup script에 넣어야 할 것

좋은 setup script는 에이전트가 중간에 헤매지 않도록 선행 작업을 몰아넣습니다.

```bash
#!/usr/bin/env bash
set -euo pipefail

npm ci
npm run build:types
python -m pip install -r requirements-dev.txt
./scripts/bootstrap-test-db.sh
```

setup에 넣을 가치가 큰 작업:
- 의존성 설치
- 런타임 버전 고정
- 네이티브 모듈 사전 컴파일
- 테스트 데이터 시드
- linter, formatter, codegen 도구 설치

반대로 setup script가 feature 전체를 대신 구현하려고 들면 좋지 않습니다.

## AGENTS.md가 샌드박스를 더 똑똑하게 만든다

Cloud Codex는 `AGENTS.md`를 보고 프로젝트 전용 명령과 규칙을 찾습니다.

따라서 좋은 `AGENTS.md`는 에이전트에게 다음을 알려줄 수 있습니다.

- 어떤 테스트 명령이 진짜 중요한지
- 어떤 lint target이 기준인지
- 어떤 생성 폴더는 건드리면 안 되는지
- 어디가 실제 source of truth인지

결국 핵심은 **샌드박스 + 지시문** 조합입니다. 환경은 **능력 경계**를 만들고, `AGENTS.md`는 **판단 경계**를 만듭니다.

## 인터넷 접근은 메스처럼 써라

OpenAI는 agent internet access를 켜면 위험이 커진다고 명시합니다.

- 신뢰되지 않은 페이지에서의 prompt injection
- 코드나 시크릿 유출
- 위험한 의존성 다운로드
- 라이선스 오염

꼭 켜야 한다면 가장 좁게 여세요.

- **domain allowlist** 사용
- 가능하면 **`GET`, `HEAD`, `OPTIONS`** 정도로 제한
- 글로벌이 아니라 **환경별**로 설정

GitHub와 패키지 레지스트리만 필요하다면 인터넷 전체를 열 이유가 없습니다.

## 시크릿과 환경 변수

클라우드 환경에서 시크릿은 사소한 설정이 아닙니다.

실무적으로는 이렇게 생각하면 좋습니다.
- setup에 꼭 필요한 빌드용 시크릿만 사용
- agent phase에는 장기 시크릿 노출 최소화
- 검증 작업은 mock 또는 테스트 자격 증명 선호

OpenAI 문서는 클라우드 환경 시크릿이 setup 단계에서만 제공되고, 기본 offline agent phase 시작 전 제거된다고 설명합니다.

## 자주 터지는 문제

### “로컬에서 계속 승인 요청이 뜬다”

대개 sandbox나 approval 설정이 너무 빡빡합니다.

**해결:** `read-only` 대신 `workspace-write`로 올리거나, `workspace-write` + `on-request` 조합을 사용하세요.

### “클라우드 에이전트가 필요한 도구를 설치하지 못한다”

대개 setup에서 설치하지 않았기 때문입니다.

**해결:** 나중에 설치하게 두지 말고 setup script에 포함하세요.

### “내 컴퓨터에서는 되는데 cloud에서는 실패한다”

숨겨진 상태가 있었을 가능성이 큽니다. 캐시, 로컬 서비스, 추적되지 않은 env var가 대표적입니다.

**해결:** setup을 명시적이고 재현 가능하게 만드세요.

### “웹에서 가져온 위험한 지시를 agent가 따라갔다”

보통 인터넷 허용 범위가 너무 넓었습니다.

**해결:** 도메인과 HTTP 메서드를 줄이고, 신뢰 가능한 리소스만 대상으로 삼으세요.

## 현실적인 시작점

### 로컬

- sandbox mode: `workspace-write`
- approval policy: `on-request`
- writable roots: 저장소와 임시 디렉터리 정도만 허용

### 클라우드

- 결정론적인 setup script
- 정말 필요할 때만 인터넷 허용
- 인터넷이 필요하면 최소 allowlist 사용
- `AGENTS.md`에 lint, typecheck, test 명령 명시

이렇게 해야 자율성은 확보하면서도 위험을 과장 없이 관리할 수 있습니다.

## Codex vs Claude Code MCP

Claude Code는 종종 MCP를 통해 실시간 도구 연결로 능력 부족을 메웁니다.
Codex는 보통 런타임 계약을 더 엄격하게 만들어 문제를 풉니다.

| 문제 | Claude Code 패턴 | Codex 패턴 |
|------|------------------|------------|
| 외부 문서 조회 | MCP 서버 | allowlist 또는 사전 준비된 문서 |
| DB 접근 | 라이브 도구 연결 | setup, mock, 제한된 서비스 접근 |
| 로컬 안전성 | 도구별 권한 | sandbox mode + approval policy |
| 재현성 | 로컬 머신 의존성 큼 | cloud container에서 강점 |

어느 쪽이 무조건 더 낫다고 보긴 어렵습니다. Codex는 **반복 가능하고 경계가 명확한 실행**에 강하고, Claude는 **라이브 도구를 연결한 탐색형 작업**에 강합니다.
