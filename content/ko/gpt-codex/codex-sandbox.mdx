## 샌드박스는 “보안 옵션”이면서 “UX 옵션”이다

샌드박스는 단순히 막는 장치가 아닙니다.
너무 빡빡하면 승인 요청이 과도하게 뜨고, 너무 넓으면 위험이 커집니다.

결국 목적은 하나입니다.
**Codex가 대부분의 정상 작업을 멈추지 않고 처리하게 하면서, 진짜 위험한 경계만 사람에게 넘기는 것**입니다.

## 로컬과 cloud는 분리해서 생각하자

### 로컬 Codex
내 머신에서 작업하지만, sandbox mode와 approval policy로 경계를 정합니다.

### Cloud Codex
OpenAI가 관리하는 컨테이너 안에서 실행됩니다.
setup script, environment variables, 인터넷 정책이 핵심입니다.

이 둘을 섞어 생각하면 왜 막히는지 파악하기 어려워집니다.

## 로컬 sandbox mode 세 가지

Sandboxing 문서 기준 요약은 이렇습니다.

| 모드 | 의미 | 적합한 용도 |
|------|------|-------------|
| `read-only` | 파일 읽기 중심, 수정/실행은 승인 필요 | 감사, 탐색, 안전한 코드 리뷰 |
| `workspace-write` | 워크스페이스 안 수정 + 일반 명령 실행 | 대부분의 일상 개발 |
| `danger-full-access` | 샌드박스 제한 제거 | 신뢰도 높은 자동화에서만 제한적으로 |

공식 문서도 `workspace-write`를 로컬 작업의 **low-friction 기본값**으로 설명합니다.

## approval policy는 멈추는 타이밍

config 문서에서 자주 보이는 조합은 아래입니다.

```toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"
```

이 조합이 좋은 이유는 단순합니다.

- 저장소 안의 일반 작업은 계속 진행하고
- 경계를 넘는 순간만 물어봄

입문자와 대부분의 팀에게 가장 현실적인 기본값입니다.

## writable roots는 좁을수록 좋다

많은 사용자가 sandbox mode만 보고 끝내지만, 실제 사고는 **너무 넓은 writable roots**에서 자주 납니다.

좋은 기본값:

- 현재 저장소
- 필요하다면 임시 디렉터리 하나

피하고 싶은 기본값:

- 홈 디렉터리 전체
- 여러 unrelated 프로젝트 루트

Codex에 일을 맡길 때 중요한 건 “쓸 수 있느냐”보다 **어디까지 쓸 수 있느냐**입니다.

## cloud task는 setup이 계약이다

Cloud environments 문서의 흐름은 대체로 이렇습니다.

1. 컨테이너 생성 + 저장소 checkout
2. setup script 실행
3. 인터넷 접근 정책 적용
4. agent phase에서 수정과 검증

여기서 중요한 사실 두 가지:

- **setup script는 인터넷이 켜진 상태에서 실행될 수 있음**
- **agent phase 인터넷은 기본적으로 꺼져 있음**

그래서 설치, codegen, test DB bootstrap 같은 선행 작업은 setup 단계로 몰아넣는 편이 안정적입니다.

## setup script에 넣을 것과 넣지 말 것

넣을 가치가 큰 것:

- 의존성 설치
- 런타임/툴 버전 고정
- 테스트 데이터 준비
- codegen
- 네이티브 빌드 선행 작업

넣지 않는 편이 좋은 것:

- feature 자체 구현
- 장시간 인간 판단이 필요한 단계

setup은 **환경을 재현 가능하게 만드는 곳**이지, 기능을 대신 만드는 곳이 아닙니다.

## 리뷰와 샌드박스는 같이 움직인다

Review 문서에 따르면 review pane은 기본적으로 **uncommitted changes**를 기준으로 보고, 필요하면 base branch diff나 last turn changes로 범위를 바꿀 수 있습니다.

이게 중요한 이유는 샌드박스와 연결되기 때문입니다.
권한을 넓힐수록, 작업 후 review를 더 자주 해야 합니다.

즉,

- 좁은 권한 = 승인 많음, 위험 낮음
- 넓은 권한 = 승인 적음, review 책임 큼

둘은 항상 세트입니다.

## 현실적인 시작점

### 로컬 기본값

```toml
model = "gpt-5.4"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
```

그리고 writable roots는 저장소 중심으로 최소화하세요.

### cloud 기본값

- setup script를 결정론적으로 만들기
- 인터넷은 필요한 경우만 허용하기
- 시크릿은 setup에 필요한 최소치만 노출하기
- `AGENTS.md`에 진짜 검증 명령 명시하기

## 흔한 실패 패턴

### “계속 승인만 뜬다”
대개 `read-only`가 과도하거나 roots가 잘못 잡혀 있습니다.

### “로컬에서는 되는데 cloud에서 실패한다”
숨은 상태를 setup에 명시하지 않았을 가능성이 큽니다.

### “너무 빨라졌는데 불안하다”
권한만 넓히고 `/review`나 Git checkpoint 습관이 없을 때 자주 생깁니다.

## 안전한 운영 원칙

- 먼저 **좁은 권한**으로 시작하고
- 승인 피로가 실제 병목일 때만 단계적으로 넓히고
- 권한을 넓힌 만큼 review와 checkpoint를 강화하세요.

Codex에서 좋은 샌드박스 설정은 “제일 센 설정”이 아니라,
**팀이 안심하고 반복해서 쓸 수 있는 설정**입니다.
