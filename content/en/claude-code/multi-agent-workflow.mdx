## The core rule: parallelize only when independence is real

Multi-agent workflows can shrink wall-clock time, isolate noisy work, and keep your main context cleaner — but they are easy to misuse.

Anthropic's subagent docs make the tradeoff clear:

- each subagent gets its **own context window**
- background work can run **concurrently**
- results come back as **summaries**
- the main conversation is still responsible for **coordination**

They work best when the job splits into lanes that are mostly independent.

## What a Claude Code subagent actually is

A subagent is a separate Claude instance with its own:

- context window
- prompt / role
- tool restrictions
- model selection
- permission behavior inherited from the parent session

That separation is the point: keep verbose, local, or repetitive work out of the main thread.

## Why this matters: context is the scarce resource

Anthropic's Claude Code best practices emphasize that context fills quickly and quality drops as it gets crowded. Subagents are one answer to that problem.

Good uses:

- large codebase exploration
- running noisy checks and returning only the failures
- comparing multiple approaches in parallel
- researching several modules at once

Bad uses:

- editing one obvious file
- any task that needs constant back-and-forth with you
- workflows where every step depends on the previous step's output

Parallelism is not the goal. **Protecting attention** is.

## Single conversation vs subagents vs team/worker workflows

This is the decision table most people need.

| Workflow shape | Best tool | Why |
|---|---|---|
| quick bug fix, one-file edit, iterative pair-programming | main conversation | shared context matters more than separation |
| independent research or review branches that can report back | subagents | separate context + concise summaries |
| workers need to talk to each other, claim tasks, or coordinate directly | team/worker workflow | collaboration matters, not just delegation |
| repeatable scripted orchestration from code or CI | CLI / Agent SDK | explicit automation and structured control |

A simple way to think about it:

- **single conversation** = one operator doing the whole job
- **subagents** = focused specialists who report back to the lead
- **team/worker workflows** = multiple workers coordinating among themselves over shared tasks

## When subagents help most

### 1. Parallel research

Anthropic explicitly documents this pattern: independent investigations across separate areas.

```text
Research the authentication, database, and API modules in parallel using separate subagents.
Each should report the top risks and the first files I should inspect.
```

This is strong because:

- each lane is independent
- the output can be summarized
- the lead can compare results without carrying every raw file read in its own context

### 2. Isolating noisy operations

Anthropic calls out high-volume operations as a prime subagent use case.

```text
Use a subagent to run the test suite and report only:
- failing tests
- the key error excerpt
- the first likely root-cause file
```

This keeps giant command output out of the main session.

### 3. Compare-and-choose workflows

Use subagents when you want multiple candidate approaches before committing.

```text
Explore three ways to simplify the notification pipeline.
- one focused on minimal diff
- one focused on performance
- one focused on maintainability
Return tradeoffs and recommend one.
```

This is a great use of parallelism because the branches are intentionally independent.

### 4. Mixed-mode flows: research first, action second

A common pattern is:

1. subagents map the problem
2. the lead chooses the approach
3. implementation happens in the lead or in a dedicated implementation agent
4. verification runs in parallel again if useful

That pattern avoids one of the biggest mistakes: letting several writers touch the same design before anyone agrees on the direction.

## When subagents are the wrong tool

Anthropic's docs explicitly recommend staying in the main conversation when the task needs frequent back-and-forth, significant shared context across phases, or just a quick targeted change.

### Do **not** parallelize when:

- step B depends on step A's output
- the same file is likely to be edited from multiple lanes
- the work is small enough that coordination overhead dominates
- you need ongoing dialogue with the agent while it works
- latency matters more than breadth

If you cannot describe the split cleanly, you probably do not have a subagent problem yet. You have a task-design problem.

## The orchestrator pattern: how to prompt multi-agent work well

The main conversation should behave like a lead engineer or technical PM. Its job is to:

1. define the split
2. assign boundaries
3. specify what each worker returns
4. synthesize and choose the next step

A strong orchestration prompt includes five things.

### 1. The goal

```text
Goal: review the checkout stack for failure handling gaps.
```

### 2. The split

```text
Split the work:
- Agent 1: frontend checkout UI and retry states
- Agent 2: backend payment and order handling
- Agent 3: test coverage and observability gaps
```

### 3. Scope boundaries

```text
Each agent should stay inside its assigned area.
Do not modify files yet.
```

### 4. Return format

```text
Each report should include:
- top 3 findings
- first files to inspect
- open questions
- whether this area blocks the others
```

### 5. Decision rule

```text
After reports arrive, synthesize the highest-confidence next step with the smallest safe diff.
```

That is what turns “please use multiple agents” from a vibe into a workflow.

## Prompt templates for explicit subagent orchestration

### Parallel investigation template

```text
Goal:
Investigate why checkout sometimes creates duplicate orders.

Use separate subagents in parallel:
- Agent 1: inspect frontend retry behavior in src/components/checkout/
- Agent 2: inspect backend idempotency in src/api/orders/
- Agent 3: inspect tests and logs related to duplicate-order incidents

Constraints:
- read-only investigation only
- each agent should stay in its assigned area
- return concise findings, not raw logs

Report back with:
- likely root cause
- confidence level
- files I should change first
- what still needs verification
```

### Parallel implementation template

Use this only when write ownership is naturally separated.

```text
Goal:
Add a user-profile feature.

Split implementation into parallel lanes:
- Agent 1: backend endpoints in src/api/users/
- Agent 2: frontend UI in src/components/profile/
- Agent 3: database migration and schema updates

Shared constraint:
- read shared types from src/types/user.ts before editing
- do not edit files owned by another lane

Return:
- completed work
- assumptions made
- integration risks
- verification run in each lane
```

### Sequential handoff template

Use this when parallel work would collide.

```text
Use a subagent to review performance issues in src/search/.
After it reports back, use the main conversation to choose one fix.
Then use an implementation-focused subagent to make the change.
Finally, run verification in the main session.
```

This is often better than parallelism because it preserves dependency order.

## Team/worker tradeoffs: when a team beats subagents

Subagents are good when each worker can disappear after returning a summary. They are worse when workers need to coordinate continuously.

That is where team/worker workflows become attractive.

### Prefer subagents when:

- only the result matters
- tasks are self-contained
- the lead can do all synthesis
- you want lower coordination overhead

### Prefer team/worker patterns when:

- workers need direct communication
- a shared task list matters
- tasks unblock one another over time
- you want long-running collaboration across several lanes

A useful mental model:

- **subagents** are disposable specialists
- **workers** are teammates with ongoing responsibilities

If workers need to negotiate ownership, claim tasks, or challenge each other's findings, simple subagents are usually too thin.

## Background vs foreground execution

Anthropic documents that subagents can run in the foreground or in the background. That distinction matters.

### Foreground is better when:

- the task may need clarifying questions
- permissions are uncertain
- you want to watch or redirect closely
- the task is risky

### Background is better when:

- the work is long-running but bounded
- permissions are already known
- the output can be summarized later
- you want to keep using the main session while it runs

A practical pattern:

- background the noisy research or test run
- keep planning or reviewing in the lead session
- return to synthesis when the worker finishes

## Context-budget thinking for multi-agent work

Parallelism can save time, but it can still make context worse if you are careless.

### Good context-budget behavior

- give each worker a narrow assignment
- ask for concise summaries instead of raw output
- avoid having many agents all return long reports
- keep integration decisions in one place

### Bad context-budget behavior

- “investigate everything in parallel”
- five workers all reading the same directories
- returning full logs, full diffs, and full reasoning from every lane
- starting new workers before choosing which earlier findings matter

Anthropic's subagent docs warn about this directly: even parallel work can consume significant context when too many detailed results flow back into the main conversation.

## Cost and model discipline

Every subagent is its own Claude call tree.
That means multi-agent workflows can multiply spend quickly.

| Decision | Cheap version | Expensive version |
|---|---|---|
| research | fast model + read-only tools | strong model exploring the whole repo |
| implementation | one bounded worker | several overlapping workers |
| verification | targeted test/report lane | full-suite reruns from every agent |

A practical rule:

- use cheaper / faster lanes for broad search and evidence gathering
- use stronger lanes for synthesis, risky edits, or tricky reasoning
- do not create more workers than you can meaningfully steer

## Worktree and ownership thinking

Parallel writing only works when file ownership is clear.

Before spawning multiple writing lanes, decide:

- which files are shared interfaces
- which files each lane owns
- who integrates cross-cutting changes
- what happens if two lanes want the same file

If the edit boundaries are messy, use a single lead implementation or isolate work more aggressively before parallelizing.

That is why some teams pair subagents with git worktrees or branch isolation for bigger automation workflows: independence has to be real in the filesystem too, not just in the prompt.

## Practical orchestration patterns that actually work

### Pattern 1: Map → choose → implement → verify

This is the safest default for broad problems.

1. parallel research subagents map the problem
2. the lead picks one direction
3. one implementation lane executes
4. verification runs afterward

Use this for:

- debugging
- architectural cleanup
- unfamiliar codebases

### Pattern 2: Parallel lanes with a shared contract

Use when several parts can move at once if they agree on one interface.

Example:

- backend defines or confirms response shape
- frontend builds against that contract
- tests prepare coverage against the same contract

The critical move is to define the shared contract **before** the workers diverge.

### Pattern 3: Reviewer + implementer loop

This is closer to a team/worker shape.

- lane A finds issues
- lane B fixes them
- lead or lane A re-checks the result

Useful for:

- security review
- performance review
- docs QA against product changes

### Pattern 4: Background heavy work + foreground decision-making

One of the most practical setups is:

- background a long test run, log analysis, or code search
- keep using the main session for lightweight planning
- synthesize only after the noisy worker finishes

This gets the benefits of parallelism without turning the whole task into a coordination puzzle.

## Common failure modes

| Failure mode | What happens | Better move |
|---|---|---|
| unclear split | agents duplicate effort or miss coverage | assign directories / responsibilities explicitly |
| overlapping write scope | merge pain or conflicting edits | assign file ownership or stay single-threaded |
| too many workers | more coordination than progress | use fewer, better-bounded lanes |
| parallelizing dependencies | workers block on each other | run the dependency chain sequentially |
| verbose returns | main context gets flooded anyway | require concise summaries |
| using subagents for collaboration | workers cannot coordinate well enough | move to a team/worker workflow |

## SDK and CLI orchestration

Claude Code is not only interactive. Anthropic also exposes programmatic surfaces for automation.

### CLI headless mode

From the CLI docs, `claude -p` / `--print` is the basic building block for non-interactive runs.

```bash
claude -p "Audit src/auth/ for auth bypass risks and output markdown"
```

Useful options from the CLI docs include:

- `--output-format json` for machine-readable output
- `--json-schema` for validated structured results
- `--max-turns` to cap agentic depth
- `--resume` to continue prior work instead of restating everything

That makes headless Claude useful for:

- CI review steps
- scripted content pipelines
- batch analysis across directories or repos

### Programmatic orchestration with the Agent SDK

The Agent SDK is the right layer when you want orchestration logic in code rather than in an interactive session.

Good fits:

- batch jobs over many repos
- reusable internal tools
- synthesis pipelines with structured outputs
- workflows that need logging, branching, or post-processing

In practice, think of the SDK as the tool for **building your own orchestrator**, while Claude Code subagents are the tool for **delegating within a live working session**.

## Cost-aware and quality-aware delegation checklist

Before launching multiple agents, ask:

- Are the lanes genuinely independent?
- Do I know what each lane owns?
- Does each worker have a concise deliverable?
- Will the main session still be able to synthesize the results cleanly?
- Would one plan-first pass reduce wasted parallel work?
- Am I parallelizing because it helps, or because it feels advanced?

If you cannot answer those confidently, stay simpler.

## Codex comparison

Codex and Claude both support delegation, but the practical feel is different.

Claude's subagent model is strongest when you want:

- focused specialists inside one broader session
- explicit tradeoffs between main-context work and isolated-context work
- interactive orchestration with selective background execution

Codex-style workflows often emphasize sharper task handoff, stronger sandbox framing, and repo-level procedural guidance. Claude's advantage is the ease of moving between direct conversation, delegated subagents, and longer-running orchestration patterns inside the same environment.

## The real maturity move

Beginner multi-agent usage sounds like:

> “Use more agents.”

Mature multi-agent usage sounds like:

> “Split the work so each lane owns a clean slice, returns a concise result, and only parallelizes what is truly independent.”

That is the difference between orchestration as spectacle and orchestration as engineering leverage.
