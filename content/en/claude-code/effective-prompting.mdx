## The core rule: write prompts like executable engineering tickets

Claude Code is most useful when it can act, verify, and iterate instead of guessing what you meant.

Anthropic's docs keep returning to four habits:

- **be explicit** about what you want
- **point Claude to the right context** instead of making it search blindly
- **give it a way to verify its work**
- **keep context clean enough** that it can still reason well late in the session

A strong prompt feels less like “chatting with an AI” and more like handing a teammate a well-scoped issue.

## The simplest prompt shape that scales

For most coding work, you can start with a four-part structure:

```text
Goal:
What should change?

Context:
Which files, symptoms, patterns, or references matter?

Constraints:
What must stay unchanged?

Done when:
How should Claude verify success?
```

That structure works because every part removes a different kind of ambiguity:

| Part | What it prevents |
|---|---|
| Goal | Claude solving the wrong problem |
| Context | wasted exploration and wrong starting assumptions |
| Constraints | scope creep and risky “helpful” changes |
| Done when | premature completion |

## What good prompts actually contain

### 1. A concrete target state

Weak:

```text
Improve the auth flow.
```

Better:

```text
Make session-timeout login failures recover automatically by fixing token refresh in src/auth/.
```

The improvement is that the second version names a **behavior**, not just a product area.

### 2. Starting points, not background essays

Anthropic recommends specific context. In practice, give Claude:

- the first file or directory to inspect
- the existing implementation pattern to copy
- the reproduction steps or failing symptom
- the most important edge case

```text
Context:
- Bug shows up after session timeout
- Start in src/auth/login.ts and src/auth/refresh.ts
- Follow the retry pattern already used in src/api/client.ts
- Write a failing test before fixing it
```

That is dramatically better than a paragraph of vague backstory.

### 3. Constraints that change decisions

Constraints matter most when they remove tempting but wrong options:

```text
Constraints:
- do not change the public API shape
- do not add dependencies
- do not touch the OAuth flow
- keep the diff inside src/auth unless a shared helper already exists
```

Good constraints are not bureaucracy. They keep Claude from “solving” the task by changing something you were relying on.

### 4. Verification Claude can run itself

Anthropic's Claude Code best-practices guide is blunt: giving Claude a way to verify its work is one of the highest-leverage moves you can make.

```text
Done when:
- login succeeds after session timeout
- the regression test passes
- auth-related typecheck stays clean
```

Without this, you become the only feedback loop.

## Prompting patterns that work in real sessions

### 1. Direct execution for small, obvious tasks

Use this when the diff is easy to describe in one sentence.

```text
Fix the TypeScript error in src/auth/login.ts.
Keep the change minimal and run the relevant tests afterward.
```

Good for:

- a single bug
- a focused docs update
- one-file refactors
- obvious type or lint fixes

Anthropic's guidance on plan mode is useful here: if you could already describe the diff in one sentence, don't add planning overhead unless the task is risky.

### 2. Explore first, then implement

Use this when you know the symptom but not the cause.

```text
Users report duplicate cart entries when they click quickly.
Explore the cart update flow first and identify the root cause.
Then implement the smallest safe fix.
Write or update a regression test.
```

This works well because it matches Claude Code's natural loop:

1. inspect
2. form a hypothesis
3. change code
4. verify

### 3. Plan first for architectural or multi-file work

Ask for a plan before any edits when:

- the task crosses multiple modules
- the rollback cost is high
- there are several plausible designs
- you are unsure which pattern in the codebase should dominate

```text
I want to add WebSocket-based notifications.
Before writing code, propose a plan covering:
- where the server integration should live
- how auth should work
- how to keep the rollout reversible
- what should be tested first
```

This is not about making Claude slower. It is about failing earlier and more cheaply.

### 4. Chain prompts instead of overloading one prompt

Anthropic's prompt-engineering docs recommend chaining for complex tasks that have distinct phases. That matters in Claude Code too.

A good chain looks like:

1. **map the problem**
2. **choose an approach**
3. **implement**
4. **review / verify**

For example:

```text
Step 1: inspect the notification pipeline and identify where retries can duplicate sends.
Step 2: propose the smallest fix.
Step 3: after I approve the approach, implement it.
Step 4: run the relevant tests and summarize any residual risk.
```

If Claude keeps dropping a step, that usually means the step should become its own prompt.

### 5. Use examples when format or tone matters

Few-shot examples are especially helpful when you care about:

- output structure
- naming conventions
- review style
- content tone for docs

```text
Write release notes in this format:

<example>
## Fixed
- Corrected race condition in checkout retry flow

## Changed
- Simplified auth logging around token refresh
</example>

Now write release notes for the current diff.
```

Anthropic's prompting guide recommends a handful of relevant examples rather than a huge example dump. Three good examples beat ten noisy ones.

### 6. Use lightweight XML-style structure for complicated asks

Claude handles plain English well, but XML-style sections help when your prompt mixes many inputs.

```text
<goal>Add regression coverage for checkout timeout failures.</goal>
<context>
- start in src/checkout/
- reuse the test helpers in tests/helpers/payment.ts
</context>
<constraints>
- no snapshot tests
- do not change production behavior unless required to make the bug reproducible
</constraints>
<done_when>
- failing test first
- fix merged into existing checkout test suite
- relevant tests pass
</done_when>
```

You do not need XML for every prompt. Use it when structure will reduce ambiguity.

## A practical escalation ladder

When Claude is struggling, the answer usually is **not** “write an even longer prompt.” Use this escalation ladder instead:

1. **Clarify the goal** — what exactly should change?
2. **Add better context** — files, symptoms, patterns, references
3. **Add harder constraints** — what must not change?
4. **Add verification** — tests, screenshots, commands, expected output
5. **Split the task** — explore → plan → implement → verify
6. **Escalate tools** — CLAUDE.md, MCP, subagents, hooks, scripts

That is the real progression from beginner prompting to effective operator behavior.

## Context-budget thinking matters more than clever wording

Anthropic's Claude Code best-practices page emphasizes that performance degrades as context fills. That changes how you should prompt.

### Prefer targeted context over exhaustive context

Bad:

```text
Please inspect the whole repo and figure out why checkout is flaky.
```

Better:

```text
Investigate checkout flakiness.
Start in src/checkout/, tests/e2e/checkout.spec.ts, and the retry helper in src/lib/retry.ts.
Focus on failures after network timeout.
```

### Put long reference material above the ask when needed

Anthropic's long-context guidance recommends putting long reference material first and the actual ask later. In Claude Code, that usually means:

- attach or paste the relevant artifact
- keep the actual request at the end
- ask Claude to quote or cite the part it is relying on when precision matters

### Use `/compact` before the session gets sloppy

A long session is not automatically a good session.
Use `/compact` when:

- Claude keeps anchoring on stale context
- too many exploratory turns are now irrelevant
- answers are getting slower and fuzzier

### Start a new conversation when the job changes

New feature, new architectural question, or a big mental-model shift? Start fresh.
Prompt quality is not only about wording. It is also about whether the current conversation still fits the job.

## Iteration loops beat “perfect prompt” fantasies

Claude Code is strongest when you treat it as a high-speed iteration partner.

A healthy loop looks like this:

1. give a bounded task
2. inspect the result
3. correct the direction
4. add the next constraint or refinement
5. verify again

```text
Round 1:
Add login rate limiting.

Round 2:
Good. Keep the server behavior, but make the user-facing error message friendlier.

Round 3:
Now add a regression test for five failed attempts within five minutes.
```

This works because each round is:

- understandable
- reviewable
- reversible
- easier to verify than a one-shot “build the whole auth system” ask

## Failure modes that make Claude look worse than it is

| Failure mode | What it looks like | Better move |
|---|---|---|
| vague scope | Claude edits unrelated files | specify directories, files, or boundaries |
| missing success criteria | output looks plausible but is unverified | define tests, screenshots, or commands |
| overstuffed prompt | Claude misses one of seven requirements | split into chained prompts |
| too much stale context | Claude keeps following old assumptions | `/compact` or start fresh |
| hidden constraints | Claude chooses a forbidden solution | say what must not change |
| asking for explanation when you want action | Claude gives analysis only | explicitly ask it to implement and verify |

## “Tell me” versus “do it” is a real distinction

Claude will happily explain if your prompt sounds analytical.
If you want action, ask for action.

```text
Explain why this API is slow.
```

That usually gets you analysis.

```text
Profile this API, identify the bottleneck, fix whatever is safe to fix now, and report what remains.
```

That gives Claude a job, not a discussion topic.

## Practical prompt templates you can steal

### Bug fix template

```text
Goal:
Fix the race condition causing duplicate cart entries.

Context:
- repro: add the same item twice quickly
- start in src/cart/useCart.ts
- related server action: src/app/actions/cart.ts
- follow the optimistic rollback pattern in src/wishlist/

Constraints:
- no new dependencies
- do not change the API contract

Done when:
- duplicate-entry repro is fixed
- rollback still works on failure
- relevant tests pass
```

### Refactor template

```text
Goal:
Refactor the payment retry logic to reduce branching and duplicate logging.

Context:
- start in src/payments/retry.ts
- callers live in src/payments/stripe.ts and src/payments/jobs/
- preserve current behavior

Constraints:
- no public API changes
- no schema changes
- keep logs equivalent unless duplication is removed intentionally

Done when:
- retry flow is easier to follow
- tests still pass
- changed behavior, if any, is called out explicitly
```

### Docs/content template

```text
Goal:
Expand this guide so it is more practical for working engineers.

Context:
- preserve frontmatter, curriculum markers, and official links
- align claims with the cited vendor docs
- match the depth of the strongest docs in the same site section

Constraints:
- do not invent unsupported product behavior
- keep internal links intact

Done when:
- the guide adds concrete patterns, examples, and failure modes
- markdown remains valid
- diff is reviewable
```

## When to bring in other Claude Code features

Prompting is only one lever.
Escalate when the task needs a stronger operating model.

### Use CLAUDE.md when the same setup keeps repeating

If you are always restating:

- repo conventions
- test commands
- coding standards
- review expectations

…that is no longer a prompting problem. It is a missing persistent instruction surface.

### Use MCP when Claude lacks live context

If the truth lives in:

- Jira
- Slack
- internal docs
- databases
- browser-accessible systems

…stop stuffing screenshots and pasted snippets into the prompt if a tool can provide fresher context.

### Use subagents when the work is noisy but separable

If a task naturally splits into independent research lanes, delegation is better than stuffing everything into one prompt. The next guide covers this in depth.

## Conversation and CLI tactics that improve prompt quality

### Reference files directly

Claude Code performs better when you point at likely touchpoints.

```text
Read src/types/user.ts first, then add avatar_url support.
```

```text
Inspect tests/auth/session.test.ts and the refresh logic in src/auth/ before changing anything.
```

This reduces wasted search and shrinks context usage.

### Use screenshots for UI work

For visual bugs, one screenshot plus one sentence often beats ten paragraphs.

```text
[paste screenshot]
The mobile nav overlaps the CTA button at 375px width. Fix the layout without changing desktop spacing.
```

### Use `claude -p` for one-shot automation

Anthropic's CLI docs support `--print` / `-p` for non-interactive runs, which is useful when you need a single focused output rather than a session.

```bash
claude -p "summarize these test failures" < failing-tests.txt
```

```bash
claude -p --output-format json --json-schema '{"type":"array","items":{"type":"string"}}'   "list all public routes in src/app"
```

Useful when:

- scripting repetitive analysis
- generating structured output
- running one-off transformations in CI

### Resume instead of repeating setup

If the task is ongoing, `claude --resume` can be better than rewriting a huge recap prompt from scratch.

That is another form of prompt hygiene: reuse good state instead of retyping it badly.

### Model and effort selection

You do not need the strongest reasoning mode for every task.

| Task shape | Good default |
|---|---|
| small edits, formatting, quick questions | lower-cost / faster model |
| normal implementation and review | default coding model |
| architecture, tricky debugging, complex tradeoffs | stronger model or higher-effort reasoning |

If you ask for deeper thinking, say **why**:

```text
Think carefully about rollback risk and migration safety before changing this auth flow.
```

That is better than vague “be smart” phrasing.

## Final checklist for strong Claude Code prompts

Before hitting enter, ask:

- Does Claude know what should change?
- Does it know where to start?
- Does it know what must stay unchanged?
- Does it know how to verify success?
- Is this one verification-sized task, or should I split it?
- Is the current conversation still clean enough to support good work?

If you can answer all six, your prompt is probably already better than most.

## The real upgrade path

The biggest prompting improvement is not learning magic words.
It is learning to turn fuzzy requests into:

- bounded tasks
- explicit constraints
- verifiable finish lines
- manageable iteration loops

That is how you stop using Claude Code like a chatbot and start using it like an engineering partner.
