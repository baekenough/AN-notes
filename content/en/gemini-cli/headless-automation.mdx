## Headless mode is where the CLI becomes infrastructure

Interactive sessions are great for discovery. Headless mode is what turns Gemini CLI into part of your delivery system.

Use it when you want Gemini to behave like a tool in a larger pipeline:

- content QA in CI
- commit or PR review automation
- release note generation
- nightly audits for drift, broken links, or risky patterns
- editor or script integrations that need structured output

The official docs define headless mode as non-interactive execution that returns text, JSON, or JSONL instead of the full terminal UI.

## The output contract matters more than the prompt poetry

The biggest automation mistake is writing an impressive natural-language prompt and forgetting that another machine has to consume the result.

For automation, think in contracts:

1. **Input scope** — which files or directories should Gemini read?
2. **Output schema** — what keys or sections must exist?
3. **Failure behavior** — what should happen if evidence is missing?
4. **Human handoff** — is this final output or a draft for review?

A good automation prompt is often a little boring. That is a feature.

## Prefer scoped prompts over “review the repo” chaos

Bad automation prompt:

```text
Review the whole repository and tell me what matters.
```

Better automation prompt:

```text
@content @messages
Compare ko, en, and es Gemini CLI guides.
Return JSON with keys:
- missingSlugs
- frontmatterDrift
- brokenRelativeLinks
- suggestedFixes
If evidence is missing, return an empty array for that key.
```

The second prompt is dramatically easier to parse, diff, test, and trust.

## `json` and `jsonl` are for different jobs

Both structured formats are useful, but they serve different needs.

### `--output-format json`

Use this when you want one final machine-readable result:

- a CI artifact
- a review summary
- a follow-up script that reads `.response`
- a content audit report

### `--output-format jsonl`

Use this when you want the event stream itself:

- tool traces
- incremental progress updates
- live dashboards or wrappers
- debugging why an automation run behaved strangely

If you are unsure, start with `json`. Move to `jsonl` only when the stream itself is valuable.

## Non-interactive auth is part of the architecture

For headless workflows, browser-style login is the wrong mental model. The official docs recommend:

- `GEMINI_API_KEY` for most automation
- Vertex AI credentials when your org already runs on Google Cloud

This is not just about convenience. It also unlocks **token caching**, which can materially reduce cost and latency for repeated analyses over the same repository.

That means a stable review pipeline is usually built on:

- deterministic prompt
- stable file scope
- API-key or Vertex auth
- structured output

## Strong automation patterns for this repo

### 1. Locale parity checks

```bash
gemini -p "@content Compare ko, en, and es for missing Gemini CLI slugs and frontmatter drift" --output-format json
```

### 2. MDX quality review

```bash
gemini -p "@content/en Review MDX files for weak descriptions, repeated examples, and broken relative links. Return JSON." --output-format json
```

### 3. Release-note drafting

```bash
gemini -p "@git diff --staged Draft release notes for the staged changes in markdown with headings, bullets, and risks." \
  --output-format text
```

These are all useful because they produce outputs a human can still inspect quickly.

## Sandbox automation when shell execution is part of the flow

If your headless run also invokes shell actions or validation in a less-trusted environment, pair automation with sandboxing.

That is especially relevant when:

- the workflow installs dependencies
- the repo is third-party or user-generated
- scripts may trigger arbitrary commands
- you are testing a brand-new automation chain

In other words, headless mode gives you structure. Sandboxing gives you containment.

## A simple GitHub Actions shape that scales

A practical CI pattern usually looks like this:

1. checkout the repo
2. scope the input aggressively
3. run Gemini with JSON output
4. parse the result into annotations, artifacts, or summaries
5. keep a human approval step for anything high impact

That last point matters. The best early automation is usually **review-first**, not auto-merge.

## Operational rules that keep headless mode healthy

Use these rules and your automation will stay reliable longer:

- keep prompts deterministic
- minimize file scope
- request explicit schemas
- treat Gemini output as draft or evidence, not authority
- store artifacts so humans can inspect failures
- promote only proven interactive workflows into CI

If you do that, headless mode becomes a teammate. If you do not, it becomes a mysterious JSON generator that nobody trusts.
