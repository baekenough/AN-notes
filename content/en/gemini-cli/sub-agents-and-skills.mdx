## Gemini gets better when it does less in the main thread

The default temptation is to keep everything inside one conversation: research, exploration, implementation, review, cleanup. That works for small tasks, but it is a poor scaling strategy.

Gemini CLI becomes much stronger when you separate two ideas:

- **Sub-agents** do work on your behalf.
- **Skills** change how the main agent behaves.

That difference is the heart of orchestration.

## Sub-agents are workers with compressed memory footprints

A sub-agent is useful when the work is broad, repetitive, or likely to flood your main context window.

The built-in sub-agents are a good clue about the intended use cases:

- **`codebase_investigator`** for large architectural mapping and root-cause digging
- **`generalist` / `general-purpose` style helpers** for broad repetitive tasks
- **CLI help specialists** for Gemini-specific usage questions
- **browser-oriented agents** when a workflow needs web interaction rather than only local files

You are effectively saying: “Go do the noisy work elsewhere and bring me back a clean synthesis.”

That is why sub-agents are so powerful in large repositories. They preserve the leader's attention.

## Skills are behavior overrides, not extra labor

Skills are different. They do not create another worker. They swap in a more specialized operating manual for the current agent.

That is why skills are so good for jobs like:

- code review
- security review
- migration checklists
- release preparation
- internal compliance workflows

If a sub-agent is a contractor, a skill is a new professional mindset.

## When to use which

| Situation | Use a sub-agent | Use a skill |
|---|---|---|
| The task is huge and noisy | Yes | Maybe |
| The task needs a stricter procedure | Maybe | Yes |
| You want to preserve the main context window | Yes | No |
| You want the main agent to follow a specialized checklist | No | Yes |

In practice, advanced workflows often use both:

1. delegate discovery to a sub-agent
2. activate a skill for the implementation or review phase

## Practical orchestration recipes

### 1. Large repo investigation

- Start in Plan Mode.
- Ask `codebase_investigator` to map the system.
- Summarize risks.
- Return to default mode only when the change boundary is clear.

### 2. Repetitive cleanup across many files

- Use a broad worker-style sub-agent for the repetitive part.
- Keep the main session focused on reviewing diffs and deciding tradeoffs.

### 3. Specialized review

- Keep the work local to the main session.
- Activate a skill like `code-review` or a custom team skill.
- Ask for findings in a predictable format.

This split is important: noisy search goes out, judgment-heavy review stays close.

## Skill discovery and reuse are a force multiplier

The official skills docs describe multiple discovery layers: user-level, workspace-level, extension-provided, and built-in skills. That means a team can decide where shared expertise should live.

Use these layers intentionally:

- **user-level** for personal habits
- **workspace-level** for repo conventions
- **extension-level** for reusable distribution
- **built-in** for generic expert procedures

That structure is one of Gemini CLI's most underrated advantages. Teams can grow from ad-hoc prompting into a documented operating model.

## Custom skills are best when the checklist is stable

Do not turn every clever prompt into a skill. Convert only the workflows that are:

- repeated often
- hard to remember correctly
- risky when done inconsistently
- easy to verify afterward

Good examples:

- content parity audits
- internal release checklist enforcement
- API contract review
- localization QA
- docs publication readiness checks

A bad skill is vague and aspirational. A good skill has a repeatable definition of done.

## A useful pattern for AI Native Notes

For this repo, I would use orchestration like this:

1. `codebase_investigator` maps routing, i18n, and content loading when architecture questions appear.
2. A custom `locale-review` skill checks parity across `ko`, `en`, and `es`.
3. A `content-release` skill prepares update summaries, broken-link checks, and metadata review before deployment.

That combination keeps the main session strategic while moving tedious validation into reusable specialist paths.

## The main failure mode: orchestrating for show

Sub-agents and skills are exciting, so teams often overuse them.

Warning signs:

- launching sub-agents for trivial tasks
- activating skills that are just long prompts with no verification logic
- delegating work that the main thread needs immediately
- creating five specialist assets when one good `GEMINI.md` rule would do

Orchestration should remove noise, not create ceremony.
