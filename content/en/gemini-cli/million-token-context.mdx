## The headline is real, but the workflow matters more than the number

Gemini CLI's huge context window is not marketing fluff. It really changes what kind of questions you can ask.

You can do things that are awkward with smaller-context tools:

- map a large repository before editing
- reason across `src`, config, docs, and localization at once
- compare patterns across distant folders in one session
- keep higher-level architectural questions alive while reading specific files

But the best practice is **not** “paste the whole repo every time.” The best practice is to use the large window to build a better map, then spend the rest of the session deliberately.

## Use a broad → narrow → delegate loop

This is the most reliable pattern.

### 1. Broad pass

Start with a repo-scale question.

```bash
gemini "@src @content @messages Map the content architecture, locale strategy, and likely maintenance risks."
```

You are not asking for implementation yet. You are asking for a map.

### 2. Narrow pass

Once the rough map exists, focus only on risky files or unclear subsystems.

- which loader actually owns slug resolution?
- where can frontmatter drift break pages?
- which component shapes article cards?

This is where the big context window pays off. Gemini already knows the larger system and can reason locally without losing the broader thread.

### 3. Delegate or specialize

If the search space is still large, hand it to a sub-agent such as `codebase_investigator` instead of burning the main conversation history on every search turn.

## Large context does not remove the need for targeting

Even with 1M tokens, careless reading still hurts:

- latency rises
- costs rise
- answers get noisier
- the agent spends attention on files that do not matter

That is why the official tooling still matters. Use targeted inclusion, project memory, and sub-agents instead of treating the context window like infinite RAM.

## `GEMINI.md` is how you stop resending stable instructions

One of the easiest wins is moving durable instructions into `GEMINI.md`.

Examples:

- locale parity rules
- content tone rules
- frontmatter invariants
- "always verify links after MDX edits"
- repo-specific commands and terminology

This does two things at once:

1. it saves prompt budget in every session
2. it improves consistency across unrelated tasks

The more stable the rule, the less it belongs in your ad-hoc prompt.

## Token caching changes the economics of large-context work

The official docs are clear that token caching depends on the authentication path. In practice, that means repeated large-context workflows are much more attractive when you use API key or Vertex-based auth rather than a casual browser-login mindset.

If you run the same architectural audit repeatedly in CI or as a recurring maintenance task, caching can change the cost profile enough that the workflow becomes practical instead of “interesting but too expensive.”

## Prompts that scale well with big context

### Good: structural questions

```text
@src @content @messages
Explain how article metadata flows from MDX frontmatter into page rendering and SEO.
```

### Good: cross-cutting impact analysis

```text
@src @content @package.json
We plan to change content navigation. Which files, routes, and UI surfaces are likely affected?
```

### Bad: vague totalizing requests

```text
Read everything and tell me what's important.
```

The bad version forces the model to guess your decision criteria. The good versions tell it what lens to apply.

## When the big window is most valuable

You get outsized value when the task spans multiple layers:

- content + routing
- localization + SEO
- docs + generated artifacts
- scripts + repo structure
- architecture + implementation risks

That is why Gemini CLI often feels strongest in medium-to-large repos that mix code, content, and operations.

## When *not* to reach for the whole repo

Do not pay the large-context tax for tiny jobs:

- editing one article title
- fixing one typo
- changing a single component class
- answering a question about one command flag

For small work, a tight prompt is still faster and clearer.
