## Gemini CLI safety is a stack, not one switch

When people ask whether Gemini CLI is safe, they usually want one answer. The real answer is a stack of protections that solve different failure modes.

1. **Trusted folders** protect you from unknown repo-local configuration.
2. **Plan Mode** protects you from acting too early.
3. **Sandboxing** protects your machine from risky execution.
4. **Checkpointing + restore** protects your work when an edit goes sideways.

If you combine all four, Gemini CLI feels much less like “YOLO automation” and much more like a controlled engineering tool.

## Trusted folders define whether a repo gets to influence the session

Trusted folders are the first gate. They determine whether Gemini CLI should treat a workspace as something that can safely provide project-specific behavior.

In an **untrusted** workspace, the CLI intentionally limits what repo-local configuration can do. That is exactly what you want when opening:

- a customer repository you do not control
- a random open-source project
- a vendor sample app with unknown scripts
- a security incident branch that you want to inspect cautiously

The trust prompt is not ceremony. It is the moment you decide whether the repo is allowed to shape the agent's behavior.

## A practical rule: inspect first, trust second

My preferred default is:

1. open unfamiliar repos as untrusted
2. inspect the file tree, lockfiles, scripts, and config
3. trust only after I understand what local instructions and tooling are about to load

That one habit dramatically lowers the odds of getting surprised by project-specific commands, settings, or tool access.

## Plan Mode prevents premature action

Trusted folders answer “Should this repo influence the session?”
Plan Mode answers “Should the agent be allowed to change anything yet?”

For complex requests, the safest flow is:

- enter Plan Mode
- map architecture and risks
- decide the edit boundary
- leave Plan Mode only after the change strategy is solid

This is especially useful in multilingual content repos like AI Native Notes. It is easy to accidentally change slugs, frontmatter, or locale parity if you jump into mutation too early.

## Sandboxing is about execution risk, not editorial judgment

Sandboxing is the next layer. It is for moments when the agent needs to run shell commands, install dependencies, or execute code that you do not fully trust yet.

Use sandboxing when the task includes things like:

- running package managers with unknown postinstall scripts
- executing test suites in unfamiliar repos
- trying build commands from third-party samples
- validating automation steps that could touch the wider machine

A useful mental model:

- **Plan Mode** controls *whether* the agent may act.
- **Sandboxing** controls *where* risky actions happen.

Those are different controls, and you often want both.

## Checkpointing is the safety net you feel only after you need it

Checkpointing creates rollback points before write operations. That means the agent can move faster without turning every session into a disaster-recovery event.

This is most valuable when:

- a task touches many files
- you are testing prompt variants
- the repository has strict formatting or generated artifacts
- you are doing cleanup and want a fast “undo” button

The restore flow is not a replacement for Git discipline, but it is excellent for recovering from “the agent technically did the task, but this diff is not what I wanted.”

## Match the guardrail to the repo type

Here is a practical playbook:

| Situation | Trust | Plan Mode | Sandbox | Checkpointing |
|---|---|---|---|---|
| Unknown external repo | Start untrusted | Yes | Usually yes | Optional |
| Your own production repo | Trust selectively | Yes for broad tasks | For risky commands | Yes for heavy edits |
| Fast local copy edits | Trusted | Optional | Usually no | Optional |
| Automation or CI experiments | Trusted runner image | Pre-plan locally | Yes | Use Git + artifacts |

The point is not to maximize friction. It is to spend safety budget where the risk actually lives.

## Safety also improves thinking quality

There is a second-order effect here: safety features improve reasoning.

When the agent is operating in a constrained mode, prompts tend to get sharper:

- “Investigate only, no edits yet.”
- “Run this inside a sandbox.”
- “Create a checkpoint before changing content.”

Those constraints are not just guardrails. They also force better task design.

## A good working routine for this repo

For AI Native Notes specifically, I would use this pattern:

1. **Open unfamiliar content areas untrusted** if they came from external contributors.
2. **Plan first** when changing shared navigation, locale structure, or slugs.
3. **Enable checkpointing** before sweeping MDX edits.
4. **Use sandboxing** only when the task truly requires command execution or installs.

That keeps the content workflow fast without pretending every change is harmless.
