## Start with a success loop, not with every feature

The fastest way to get comfortable with Codex is not to memorize every setting.
It is to complete one small, safe task end to end.

Early on, you only need to answer four questions:

- Which client will you use?
- Which repository will you open?
- How small is your first task?
- How will you roll back if you dislike the result?

That is the beginner loop:
**pick a bounded task, give Codex clear context, review the diff, checkpoint the result, repeat.**

## Where should you start?

Codex can be used through the app, CLI, IDE extension, and cloud flows.
The official quickstart covers all of them, but they are not equal for learning.

### Codex app
Best for most people at the start.
Review, branches, and diff inspection are visible in one place.
If you want the easiest mental model for “ask → inspect → iterate,” start here.

### CLI
Great if you live in the terminal.
It is the clearest way to see prompts, commands, slash commands, and verification loops.
It is also where many configuration and sandbox details feel most explicit.

### IDE extension
Useful if you prefer to stay inside your editor.
It can feel natural once you already understand the flow, but for your first few sessions the app or CLI often makes Codex behavior easier to reason about.

### Cloud
Do not start here unless you already know what your local workflow should look like.
Cloud becomes much easier after you already understand task sizing, review, and rollback locally.

## Local first, cloud second

A good beginner sequence is:

1. succeed locally with a tiny task
2. repeat the loop a few times
3. learn review and checkpoints
4. only then move into cloud or automation

This order works because local runs are easier to explain and review, and cloud failures are easier to debug once you already know what a healthy local run looks like.
The wrong beginner pattern is trying to learn prompts, sandboxing, setup scripts, and cloud runtime behavior all at once.

## Your first 10 minutes

The Quickstart flow is straightforward:

1. Install the Codex app or CLI
2. Sign in
3. Select the project folder
4. Send your first task

If you use the CLI, the quickstart also makes it easy to point Codex at your current repo and begin with a local request.
The important beginner move is not picking the perfect tool; it is getting to the first full success loop quickly.

## Pick the repo before you pick the task

Your first Codex task should happen in a repo that is easy to understand and easy to revert.
Strong first candidates:

- a personal project you already know
- an internal docs repo
- a small service with working tests
- a side project where a small mistake is cheap

Weaker first candidates:

- the largest monorepo on your machine
- a repo with broken setup or failing tests
- a codebase where you cannot tell whether the diff is safe
- a production-critical service you do not yet understand

Your first win should teach the workflow, not test your courage.

## Size your first task around one obvious review

The best beginner tasks have three traits:

1. **small enough to inspect manually**
2. **clear enough that success is visible**
3. **safe enough that rollback is cheap**

Good first-task shapes:

- one docs page
- one failing test or one narrow test gap
- one bounded refactor in one directory
- one bug with a clear reproduction

Poor first-task shapes:

- “modernize the whole app”
- “fix everything this week”
- “clean up auth, billing, and onboarding”
- anything where you cannot name the likely verification step before you start

If you cannot explain how you will review the result, the task is still too large.

## Your first prompt should feel like work, not like a wish

Best Practices recommends giving Codex a clear structure:

- Goal
- Context
- Constraints
- Done when

A strong first prompt looks like this:

```text
Goal:
Find unused utility candidates under src/lib/.

Context:
- Start in src/lib/
- Prefer import/usage evidence over guesses

Constraints:
- Investigation only; do not modify files

Done when:
- You list the top three candidates with reasons
```

That works because it gives Codex:

- a target
- a starting point
- a boundary
- a finish line

Beginners often think they need an elaborate prompt.
Usually they need a smaller task and a clearer finish line instead.

## Use Git checkpoints early

Quickstart explicitly recommends Git checkpoints.
That is a good habit because Codex can change a lot quickly.

A beginner-friendly loop looks like this:

```bash
git status
git add -A && git commit -m "checkpoint: before codex task"
```

If the result looks good, make another checkpoint after the task.
This removes a lot of “I think this is better, but I can’t see what happened” anxiety.

### A practical checkpoint rhythm

Use checkpoints at these moments:

- before the first Codex task in a repo
- before any task that might edit multiple files
- after a result you want to preserve before trying a variation
- before asking Codex to attempt a riskier follow-up

You do not need perfect commit messages for checkpoints.
You need fast recovery.

## Review is part of the loop, not the cleanup step

A lot of beginners treat review as optional if the output “looks plausible.”
That is exactly when they should review.

The official review flow matters early because it helps you answer:

- what changed?
- which hunks do I want to keep?
- where did scope drift happen?
- what feedback should I give before the next turn?

The review pane shows the current Git diff, and `/review` can summarize issues in the working tree.
That means your loop is really:

1. ask
2. inspect
3. review
4. iterate or accept

not just “ask and hope.”

## Slash commands worth learning in the first week

The CLI exposes many slash commands, but early on you mostly need a small core set.

### `/status`
Check the current model, approval policy, writable roots, and remaining context.
When Codex behaves differently than you expected, this is often the fastest diagnostic step.

### `/init`
Generate an `AGENTS.md` scaffold in the current directory.
Use it once you get tired of re-explaining repo rules.

### `/review`
Ask Codex to review your working tree.
Use this after implementation or before committing.

### `/mcp`
See which MCP tools are available in the session.
This matters once your task depends on external docs, trackers, or systems.

### `/plan`
Use this when the task crosses multiple files or architecture boundaries and you want a plan before coding.

### `/compact`
Compress the visible conversation when the thread is getting long and noisy.

Everything else can wait until the first loop feels stable.

## A useful onboarding loop for week one

In week one, this progression works well:

### Day 1–2: read-only wins

- ask for repo explanation
- ask for file mapping
- ask for likely change points
- inspect how Codex reasons before you trust it to edit

### Day 2–3: tiny edits

- docs update
- narrow refactor
- one small test improvement
- always review the diff before you accept it

### Day 3–4: instructions and repeatability

- use `/init`
- edit the generated `AGENTS.md` so it reflects the real repo
- add build, lint, or test commands that actually work

### Day 4–5: verification habits

- ask Codex to run the relevant checks
- use `/review` again
- compare the diff to the requested scope

### End of week one: stable loop before scale

- only after the loop feels boring should you add more automation
- only after local success should you move deeper into cloud or unattended workflows

The healthy order is **task → review → rule-making → automation**, not “turn on everything.”

## Common beginner failure modes

### “Codex changed more than I expected”
Usually the task was too broad or the finish line was vague.
Shrink the task and add stronger constraints.

### “I cannot tell whether the change is good”
The task is too large, the repo is unfamiliar, or the review loop is missing.
Return to a smaller task.

### “It worked once, but I cannot reproduce the success”
You probably skipped checkpoints, instructions, or verification.
Write down the loop while it is still fresh.

### “I keep asking for giant tasks”
That is normal at first.
Translate them into one reviewable unit of work at a time.

### “Cloud feels confusing”
That usually means local mental models are not stable yet.
Go back to the local loop and make it boring first.

## First-task templates you can copy

### Read-only onboarding task

```text
Goal:
Explain this repo in five bullets.

Context:
- Focus on the main application code paths
- Call out the likely entrypoints and test locations

Constraints:
- Read-only; do not modify files

Done when:
- I have a short repo map and the first files I should inspect next
```

### Tiny bug-fix task

```text
Goal:
Fix the empty-state button label in the dashboard.

Context:
- UI likely lives under src/components/dashboard/
- Match nearby button copy conventions

Constraints:
- Keep the change minimal
- Do not change layout or behavior beyond the label

Done when:
- The label text is corrected
- Relevant checks still pass
```

### Small docs task

```text
Goal:
Improve the onboarding paragraph in docs/setup.md.

Context:
- Keep wording aligned with the surrounding docs tone

Constraints:
- Edit that file only unless a linked doc also needs a broken link fix

Done when:
- The explanation is clearer
- Links still make sense
- The diff stays small
```

## Quick checklist before your first serious session

- Am I in the right repo?
- Is the first task small enough to review manually?
- Do I know how I will roll back?
- Did I make a Git checkpoint?
- Do I know which slash command to use if I get confused?

## Fast summary

- Start with the **app or CLI on a local repo**.
- Pick a first task that is small, reviewable, and reversible.
- Use Git checkpoints from day one.
- Learn `/init`, `/review`, `/diff`, `/mcp`, and `/status` early.
- Build a boring local success loop before you scale into cloud or automation.
